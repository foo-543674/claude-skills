# テストレビュー

コードレビュー時にテスト設計・テストの質の観点で指摘すべきポイントを定義する。

## レビュー観点

### 1. テストファースト・テスト名が仕様になっているか

- **目的: テストコマンドの実行結果に、仕様を説明する文章が出力されること**
- テスト名から具体的な仕様が読み取れるか（ユースケース的な説明であること）
  - 良い例: `should reject registration when email has no at sign`
  - 悪い例: `should return error when email format is invalid`（どういう format が invalid なのかわからない）
  - 悪い例: `should success`（何がどうなれば success なのかわからない）
  - 悪い例: `should return true`（戻り値の型の説明でしかない）
- `should` / `must` の prefix はテスト出力に仕様文が現れるようにするためのもの。フレームワークが自動付与する場合（ScalaTest の `should` メソッド等）は不要
- テスト一覧を読むだけで対象モジュールの仕様が把握できるか
- テストが先に書かれ、実装がそのテストを満たす形で進められているか

### 2. テストケースの網羅性と効率化

- 正常系だけでなく異常系・境界値がテストされているか
- 代表的な境界値: 0、1、空、null、最大値、最小値、境界の前後
- 条件分岐のすべてのパスを通るテストがあるか
- エラーケース（例外発生、タイムアウト、不正入力等）のテストがあるか
- 同じ構造で入力だけが異なるテストが並んでいないか（パラメータ化テストで1つにまとめる）
- Property-based testing が有効な場面（数学的性質、変換の可逆性、不変条件等）で活用されているか

### 3. テストの構造と可読性

- Arrange-Act-Assert（AAA）パターンで構造化されているか
- 1 テスト 1 アサーション（概念的に。物理的に複数のアサートでも 1 つの振る舞いを検証していれば OK）
- テストの意図が読んですぐわかるか（テスト内の変数名、セットアップの明確さ）

### 4. テストの独立性

- テスト間に実行順序の依存がないか
- 共有状態（グローバル変数、DB の状態等）がテスト間で干渉しないか
- 各テストが自分で必要な前提条件をセットアップしているか
- teardown / cleanup が確実に行われるか

### 5. テストダブルの適切な使用

- モック / スタブの使用が適切か（外部依存の置き換えに限定）
- 実装の詳細（内部メソッドの呼び出し順序等）をモックで検証していないか
- モックが多すぎないか（モックだらけのテストは設計の問題を示唆）
- テストダブルの振る舞いが実際の依存先と乖離していないか

### 6. テストの保守性

- テストが実装の変更に過度に敏感になっていないか（振る舞いをテストする）
- テストヘルパーやファクトリが適切に使われているか（セットアップの重複削減）
- ただしテストヘルパーの抽象化が過度で、テスト自体が読みにくくなっていないか
- マジックナンバーがテスト内に埋まっていないか（意図が伝わる値を使う）

### 7. テストの信頼性

- フレイキーテスト（実行するたびに結果が変わる）の要因はないか
  - 時刻依存、乱数依存、外部サービス依存、並行処理のタイミング依存
- テストが本当に失敗すべきとき失敗するか（アサーションが甘すぎないか）
- テスト対象のコードを意図的に壊したとき、テストが検出できるか

### 8. テストの粒度と種類（テストピラミッド）

テストピラミッドを意識し、下層ほど多く・上層ほど少なく保つ:

- **ユニットテスト（最多）**: ロジックの網羅、境界値、エッジケースはここで担保する
- **統合テスト（中程度）**: コンポーネント間の結合・連携の正常動作を確認する。ユニットテストで網羅済みのエッジケースを統合テストに重複して書かない
- **E2E テスト（最少）**: ユーザーシナリオの主要パスのみ。壊れやすく遅いので最小限に

確認ポイント:
- カバレッジを意識しすぎて、単体テストで十分なエッジケースを統合テストや E2E に書いていないか
- 各テストが適切な層に配置されているか（結合の確認なら統合テスト、ロジックの確認ならユニットテスト）
- 上位層のテストが失敗したとき、原因箇所を特定しづらい構造になっていないか（下位層のテストで先に検出できるのが理想）

## 指摘の出し方

- 「○○のケースがテストされていない。この入力でバグが起きうる」のように具体的なケースを示す
- テストの書き方の指摘は、可読性・保守性の向上に直結するものに絞る
- 「テストを増やせ」ではなく「○○の境界値テストを追加すると安心」のように具体的に提案する
