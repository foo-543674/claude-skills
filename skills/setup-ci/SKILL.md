# CI/CD パイプライン構築

自動テスト・静的解析・デプロイのパイプラインを構築し、技術品質を継続的に担保する仕組みを作る。

技術品質の基盤。製品コードを書く前に整備する。

## Why: なぜ CI/CD パイプラインが重要か

CI/CD パイプラインの品質は、技術品質の継続性・開発速度・リリース頻度を根本的に決定する。CI/CD なしでは、品質チェックが属人化し、リリースが恐怖になり、デグレードが多発する。

**根本的な理由**:
1. **品質チェックの自動化と強制**: 人間の手動チェックは、忘れ・見落とし・疲労により不完全。CI/CD により、テスト・lint・型チェックを自動実行し、品質チェックを強制する。マージブロックにより、品質が低いコードの混入を構造的に防ぐ。品質チェックの自動化なしでは、品質が属人化し、チームの規模拡大とともに品質が低下する
2. **フィードバックループの高速化**: CI/CD により、コード変更の影響が数分で検出される。手動テストでは、影響の検出に数日かかり、修正コストが膨大になる。フィードバックが速ければ、問題の原因が明確で、修正が容易。遅ければ、原因の特定が困難で、修正コストが指数関数的に増大する
3. **リリースの安全性と頻度の向上**: CD により、デプロイが自動化・標準化され、人為的ミスが排除される。手動デプロイは、手順ミス・環境差異・ロールバック困難により、リリースが恐怖になる。CD があれば、リリースが安全になり、リリース頻度が上がり、価値提供のサイクルが加速する
4. **チームの開発効率の向上**: CI/CD により、コードレビューで品質チェックの負担が減り、本質的な設計・ロジックのレビューに集中できる。手動チェックでは、レビュアーが lint・テストの実行を手動で行い、レビューコストが膨大になる。CI/CD は、チームの時間を価値創造に集中させる
5. **技術的負債の早期発見**: CI でカバレッジ・複雑度・依存関係等を継続的に監視すれば、技術的負債の蓄積を早期に発見できる。CI なしでは、技術的負債が蓄積し続け、修正不可能な規模に達してから発覚する。早期発見なら、小さな修正で対応できる

**CI/CD の目的**:
- 品質チェックを自動化・強制し、品質の継続的担保を実現する
- フィードバックループを高速化し、修正コストを最小化する
- リリースを安全化・頻度向上し、価値提供サイクルを加速する
- チームの開発効率を向上し、価値創造に集中させる
- 技術的負債を早期発見し、修正可能な段階で対応する

## プロセス

### 1. 現状の確認

**確認すべきこと**:
- 既存の CI/CD ツール（Jenkins, CircleCI, GitHub Actions 等）の有無
- 既存のパイプライン定義ファイル
- テスト実行コマンド、lint コマンド、ビルドコマンド
- デプロイ先・デプロイ方法
- シークレット管理の方法

**既存ツールがある場合**: そのツールに従う。別ツールへの移行は提案しない。

**既存ツールがない場合**: GitHub Actions をベースに構築する。

### 2. パイプライン設計

#### 2.1 CI パイプライン（プルリクエスト時）

技術品質を自動検証するゲートとして設計する。

**必須ジョブ**:

| ジョブ | 目的 | 失敗時 |
|-------|------|--------|
| Lint | コードスタイルの統一 | マージブロック |
| Type Check | 型安全性の担保 | マージブロック |
| Unit Test | ビジネスロジックの正しさ | マージブロック |
| Integration Test | コンポーネント間の連携 | マージブロック |

**任意ジョブ**（プロジェクトの要件に応じて）:

| ジョブ | 目的 | 備考 |
|-------|------|------|
| E2E Test | ユーザーシナリオの検証 | 実行時間が長い場合は main マージ後に実行 |
| Security Scan | 依存パッケージの脆弱性検出 | Dependabot / Trivy 等 |
| Coverage Report | テストカバレッジの可視化 | 数値をゲートにするかはチーム判断 |
| Build | ビルド可能性の確認 | デプロイ対象がある場合 |

#### 2.2 CD パイプライン（マージ後）

**設計原則**:
- main ブランチへのマージをトリガーにする
- ステージング → 本番の段階的デプロイ
- ロールバック手順を明確にする
- デプロイ成功の判定基準を定義する（ヘルスチェック、スモークテスト等）

### 3. GitHub Actions の実装

#### 3.1 ワークフローファイルの構成

```
.github/
└── workflows/
    ├── ci.yml        # PR 時の CI
    └── cd.yml        # マージ後の CD（必要な場合）
```

#### 3.2 CI ワークフローの設計方針

```yaml
# .github/workflows/ci.yml の構成イメージ
name: CI
on:
  pull_request:
    branches: [main]

jobs:
  lint:
    # formatter / linter の実行
  type-check:
    # 型チェックの実行
  test:
    # テストの実行
    # カバレッジレポートの出力（任意）
```

**設計上の判断ポイント**:
- ジョブの並列化: lint, type-check, test は独立して並列実行する
- キャッシュ: 依存パッケージのキャッシュで実行時間を短縮する
- マトリクス: 複数バージョン対応が必要な場合に使う（ライブラリ等）
- タイムアウト: 各ジョブに適切なタイムアウトを設定する

#### 3.3 ブランチ保護の設定

CI パイプラインの効果を担保するため、ブランチ保護ルールの設定を提案する。

**推奨設定**:
- CI の全ジョブがパスしないとマージできない
- レビュー承認が必要（チーム開発の場合）
- force push の禁止

### 4. 動作確認

パイプライン構築後、以下を確認する。

- [ ] CI ワークフローが PR 作成時にトリガーされる
- [ ] 各ジョブが正常に実行・完了する
- [ ] テスト失敗時にジョブが失敗する（意図的に壊して確認）
- [ ] lint エラー時にジョブが失敗する
- [ ] キャッシュが効いている（2 回目以降の実行時間が短縮される）
- [ ] CD がある場合、マージ後にデプロイが実行される

## Jenkins 等の既存ツールがある場合

- 既存の Jenkinsfile / パイプライン定義を読み取り、上記の必須ジョブが含まれているか確認する
- 不足しているジョブがあれば追加を提案する
- ツール固有の設定・プラグインに従う
- GitHub Actions への移行は、明示的に求められない限り提案しない

## よくあるアンチパターン

- **CI Without Teeth**: CI が動いているがマージブロックになっておらず、失敗しても無視してマージできる
- **Slow CI**: テストの並列化やキャッシュを活用せず、CI の実行に時間がかかりすぎる
- **Flaky Tests in CI**: CI 上でのみ不安定に失敗するテストを放置する
- **No Local Reproducibility**: CI でしか実行できないチェックがあり、ローカルで事前確認できない
- **Secret Sprawl**: シークレットが多数のワークフローに散在し、管理が困難になる
