# 要件の引き出し・整理

曖昧な要求や漠然としたアイデアから、実装可能な粒度の要件を引き出して整理する。

このスキルのゴールは「何を作るか」を明確にすること。「どう作るか」は設計フェーズに委ねる。

## Why: なぜ要件の引き出しが重要か

要件の引き出しの品質は、プロジェクトの成否・開発効率・顧客満足度を根本的に決定する。曖昧な要件のまま実装に入ると、手戻り・仕様齟齬・無駄な機能が大量発生し、プロジェクトが失敗する。

**根本的な理由**:
1. **手戻りコストの削減**: 曖昧な要件のまま実装すると、実装後に「これじゃない」と判明し、大量の手戻りが発生する。実装フェーズの修正は、要件フェーズの修正の 10倍〜100倍のコストがかかる。適切な要件引き出しにより、実装前に曖昧さを解消し、手戻りコストを最小化できる
2. **仕様齟齬の防止**: ユーザーの要求と実装者の理解が食い違うと、作ったものが使われない・価値を生まない結果になる。要件引き出しにより、背景・目的を理解し、暗黙の前提を明示化し、ステークホルダー間の認識を一致させる。齟齬が放置されると、プロジェクト全体が無駄になる
3. **優先順位の明確化**: すべての要求を実装すると、スコープが際限なく膨らみ、プロジェクトが破綻する。要件引き出しにより、背景・目的を理解し、本質的に重要な要件と、Nice to have を区別できる。優先順位が明確なら、限られたリソースで最大の価値を生む機能に集中できる
4. **設計品質の決定**: 要件が曖昧だと、設計が実装者の推測に依存し、後から要件が明確になった時点で設計が適合しない。要件が明確なら、最適な設計を最初から選択できる。設計の後戻りは、実装の後戻り以上にコストが高い
5. **非機能要件の見落とし防止**: 明示されない非機能要件（パフォーマンス、セキュリティ、可用性等）は、後から追加が極めて困難。要件引き出しで非機能要件を明示化しないと、リリース直前に「遅い」「落ちる」「危険」が発覚し、プロジェクトが炎上する

**要件引き出しの目的**:
- 実装前に曖昧さを解消し、手戻りコストを最小化する
- ステークホルダー間の認識を一致させ、仕様齟齬を防止する
- 本質的に重要な要件を特定し、優先順位を明確にする
- 最適な設計を選択できる明確な要件を提供する
- 非機能要件を見落とさず、後工程での炎上を防ぐ

## 判断プロセス（決定ツリー）

要件引き出し時は、以下の順序で判断する。背景・目的の理解から始め、段階的に具体化する。

### ステップ1: 背景・目的の理解（最優先）

**判断基準**:
要求の言葉をそのまま要件にしない。背後にある目的・動機を把握し、本質的な課題を特定する。

**チェック項目**:

1. **5W1Hの確認**:
   - **誰が（Who）**: 使うのか（ユーザー種別、ロール、権限レベル）
   - **なぜ（Why）**: 必要なのか（解決したい課題、達成したい目標）
   - **いつ（When）**: 必要なのか（優先度、期限、フェーズ分け）
   - **現状（What/How）**: はどうなっているか（既存の仕組み、ワークアラウンド、不満点）

2. **本質的な課題の特定**:
   - 表面的な要求: 「検索機能がほしい」
   - 本質的な課題: 「特定の条件でデータを素早く見つけたい」
   - → 検索以外の解決策もありうる（フィルタ、ソート、お気に入り等）

**判定フロー**:
```
ユーザーの要求を受ける
  ↓
なぜそれが必要なのか？
  ↓
解決したい課題を特定
  ↓
他の解決策はないか？
  ↓ ある → 複数の選択肢を提示
  ↓ ない → 要求された解決策を採用
```

**判定**: 背景・目的を理解せずに要件化しない

### ステップ2: 機能要件の洗い出し（高優先度）

**判断基準**:
CRUD（Create, Read, Update, Delete）をベースに、漏れなく機能を列挙する。

**チェック項目**:

1. **CRUDの網羅**:
   - 作成（Create）: 新規データの登録
   - 参照（Read）: データの検索・表示
   - 更新（Update）: 既存データの変更
   - 削除（Delete）: データの削除

2. **各アクションの詳細**:
   - 入力と期待される出力
   - 前提条件・事前条件
   - 結果として変化するシステムの状態
   - 関連する他の機能への影響

**判定フロー**:
```
機能要件を列挙
  ↓
CRUDを網羅しているか？
  ↓ No → 不足しているアクションを追加
  ↓ Yes
  ↓
各アクションの前提条件・出力が明確か？
  ↓ No → 詳細を確認
  ↓ Yes → OK
```

**判定**: CRUD観点で機能の漏れを防ぐ

### ステップ3: 非機能要件の引き出し（高優先度）

**判断基準**:
明示されにくい非機能要件を漏れなく確認する。プロジェクトの文脈で重要なカテゴリに絞る。

**チェック項目**:

1. **主要カテゴリ**:
   - パフォーマンス: 想定データ量、同時アクセス数、応答時間
   - セキュリティ: 認証・認可、データの機密レベル、監査ログ
   - 可用性: 許容ダウンタイム、障害時の振る舞い
   - 互換性: 対応ブラウザ・デバイス、外部システム連携
   - 運用: 監視・アラート、ログ要件
   - 制約: 技術スタック、ライセンス、法規制

2. **優先度の判断**:
   - すべてのカテゴリを網羅する必要はない
   - プロジェクトの文脈で重要なカテゴリに絞る

**判定フロー**:
```
非機能要件のカテゴリを確認
  ↓
プロジェクトで重要なカテゴリは？
  ↓
パフォーマンス重要？ → 想定データ量・応答時間を確認
セキュリティ重要？ → 認証・認可・機密レベルを確認
可用性重要？ → ダウンタイム・障害時振る舞いを確認
```

**判定**: 明示されにくい非機能要件を漏れなく確認

### ステップ4: スコープの定義（中優先度）

**判断基準**:
やること・やらないことを明確に線引きし、スコープクリープを防ぐ。

**チェック項目**:

1. **In Scope / Out of Scope / Future**:
   - **In Scope**: 今回対応する機能・要件
   - **Out of Scope**: 意識的に今回は対応しない項目（理由付き）
   - **Future Consideration**: 将来的に対応する可能性があるが、今回は設計時に考慮だけする項目

2. **優先順位付け**:
   - Must Have（必須）
   - Should Have（望ましい）
   - Could Have（できれば）
   - Won't Have（今回は対応しない）

**判定フロー**:
```
すべての要件を列挙
  ↓
優先順位を付ける（Must/Should/Could/Won't）
  ↓
Must Haveのみ In Scope
Should Have以下は Out of Scope or Future
```

**判定**: スコープを明確に定義し、無限に膨らむのを防ぐ

### ステップ5: 前提条件とリスクの整理（低優先度）

**判断基準**:
要件が成り立つための前提条件と、実現を阻む可能性のあるリスクを列挙する。

**チェック項目**:

1. **前提条件**:
   - この要件は○○が成り立つことを前提としている
   - 依存関係: 他チーム・他システムへの依存

2. **リスク**:
   - 技術的不確実性
   - 要件の変更可能性
   - 外部要因（法規制、ベンダーサポート等）

3. **未決事項**:
   - まだ決まっていないが、設計・実装前に決める必要がある項目

**判定**: 前提条件・リスクを明示化し、後工程での齟齬を防ぐ

## プロセス

### 1. 要求の背景を理解する

ユーザーが述べた要求の背後にある目的・動機を把握する。

- **誰が** 使うのか（ユーザー種別、ロール、権限レベル）
- **なぜ** 必要なのか（解決したい課題、達成したい目標）
- **いつ** 必要なのか（優先度、期限、フェーズ分けの必要性）
- **現状** はどうなっているか（既存の仕組み、ワークアラウンド、不満点）

要求の言葉をそのまま要件にしない。「検索機能がほしい」の裏には「特定の条件でデータを素早く見つけたい」という目的があり、検索以外の解決策がありうる。

### 2. 機能要件を洗い出す

要求を具体的な機能に分解する。

**洗い出しの観点**:
- ユーザーが実行するアクション（CRUD ベースで漏れを確認）
- 各アクションの入力と期待される出力
- アクションの前提条件・事前条件
- アクションの結果として変化するシステムの状態
- 関連する他の機能への影響

**粒度の判断基準**:
- 1 つの要件が 1 つの受け入れ基準セットで検証できるか
- 要件を分割したとき、片方だけリリースして意味があるか
- 1 要件が複数の異なるユーザーストーリーにまたがっていないか

### 3. 非機能要件を引き出す

明示されにくいが設計・実装に大きく影響する要件を漏れなく確認する。

| カテゴリ | 確認すべきこと |
|---------|--------------|
| パフォーマンス | 想定データ量、同時アクセス数、応答時間の期待値 |
| セキュリティ | 認証・認可要件、データの機密レベル、監査ログの必要性 |
| 可用性 | 許容ダウンタイム、障害時の振る舞い、データ復旧要件 |
| 互換性 | 対応ブラウザ・デバイス、外部システム連携、データ移行 |
| 運用 | 監視・アラート要件、ログ要件、メンテナンス手順 |
| 制約 | 技術スタック制約、ライセンス制約、法規制 |

すべてを網羅する必要はない。プロジェクトの文脈で重要なカテゴリに絞る。

### 4. スコープを定義する

やること・やらないことを明確に線引きする。

- **In Scope**: 今回対応する機能・要件のリスト
- **Out of Scope**: 意識的に今回は対応しない項目（理由付き）
- **Future Consideration**: 将来的に対応する可能性があるが、今回は設計時に考慮だけする項目

スコープの判断でトレードオフが発生した場合は ADR を作成する。

### 5. 前提条件とリスクを整理する

要件が成り立つための前提条件と、実現を阻む可能性のあるリスクを列挙する。

- **前提条件**: この要件は○○が成り立つことを前提としている
- **依存関係**: 他チーム・他システムへの依存
- **リスク**: 技術的不確実性、要件の変更可能性、外部要因
- **未決事項**: まだ決まっていないが、設計・実装前に決める必要がある項目

## 成果物

```markdown
# 要件整理: [機能名]

## 背景・目的
[誰が、なぜ、この機能を必要としているか]

## 機能要件
1. [要件1]: [具体的な説明]
2. [要件2]: [具体的な説明]

## 非機能要件
- [カテゴリ]: [具体的な要件]

## スコープ
- **In**: [対応する項目]
- **Out**: [対応しない項目と理由]
- **Future**: [将来検討する項目]

## 前提条件
- [前提1]

## リスク・未決事項
- [リスク/未決事項と対応方針]
```

## ADR の作成基準

以下に該当する判断を行った場合、ADR を作成する:

- スコープの取捨選択で複数の選択肢があり、トレードオフを伴う判断をした場合
- 非機能要件の水準設定で、コストや開発期間とのトレードオフがある場合
- 要件の優先順位付けで、ステークホルダー間の利害が対立する場合

ADR のテンプレートと配置場所はプロジェクトの既存方針に従う。プロジェクトに ADR の慣習がない場合は `docs/adr/` に連番で配置し、MADR 形式を提案する。

## よくある失敗パターン

- **Solution First**: 要求の背景を理解せず、いきなり技術的な解決策から入る
- **Scope Creep**: スコープを明確にせず、議論の中で要件が際限なく膨らむ
- **Happy Path Only**: 正常系の要件だけ洗い出し、異常系・エッジケースを見落とす
- **Implicit Assumption**: 前提条件を暗黙のまま放置し、後工程で齟齬が発生する
- **Gold Plating**: ユーザーが求めていない過剰な要件を追加する
