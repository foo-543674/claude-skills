# レビューレポート生成

各レビュースキルで得られた指摘を1つのレポートに統合する。

## Why: なぜレビューレポート生成が重要か

レビュー結果の統合と優先順位付けは、限られたリソースで最大の品質向上を実現するために必須である。

**根本的な理由**:
1. **優先順位の明確化**: 複数の観点からの指摘を重大度順に整理しなければ、「何から対応すべきか」が不明瞭になる。Critical な問題（セキュリティ脆弱性、データ損失リスク等）を放置して、Nit レベルの改善に時間を費やすリスクがある
2. **技術品質と製品品質の区別**: 技術品質（セキュリティ、テスト等）は「当然担保されている」と期待されるもので、欠けたときの影響が広範囲に波及する。製品品質（命名、可読性等）よりも優先すべきだが、区別せずに並列に扱うと、重要な問題が埋もれる
3. **レビュー観点の包括性**: 1つの観点のみでレビューすると、他の重大な問題を見逃す。複数観点を包括的に確認し、結果を統合することで、品質の穴を減らす
4. **既存コードへの敬意**: 既存コードは過去に価値を生んできたものであり、無関係な設計問題を要修正として扱うと、変更範囲が広がり、リスクが増大する。変更スコープを明確にし、関連する問題のみを指摘することが重要
5. **レビューイの負担軽減**: 指摘だけを羅列すると、レビューイは否定された印象を受ける。Good Points を明示することで、建設的なフィードバックとなり、学習効果も高まる

**レビューレポート生成の目的**:
- Critical な問題を最優先で修正させる
- 技術品質と製品品質の優先順位を明確にする
- 複数観点の包括的なレビューで品質の穴を減らす
- 変更スコープを尊重し、無関係な問題を要修正としない
- 建設的なフィードバックで学習効果を高める

## 判断プロセス（決定ツリー）

レビューレポート生成時は、以下の順序で判断する。影響範囲が大きく、優先度が高い問題から整理する。

### ステップ1: 変更スコープの確認（最優先）

**判断基準**:
レビュー対象の変更目的とスコープを把握し、関連しない問題を要修正として扱わない。

**チェック項目**:

1. **変更の目的と範囲の特定**:
   ```
   ✅ 新機能追加 → 追加部分に関連する問題を指摘
   ✅ 不具合修正 → 修正箇所に関連する問題を指摘
   ✅ hotfix → 修正箇所のみ、設計問題は指摘しない
   ✅ リファクタリング → 広範囲の設計問題も指摘可能
   ```

2. **既存コードの扱い**:
   ```
   ❌ 変更していない既存コードの設計問題を Critical/Warning で指摘
   ✅ 既存コードの問題は Note として言及し、別 Issue を提案
   ```

**判定フロー**:
```
指摘を確認
  ↓
変更されていないコードへの指摘か？
  ↓ Yes → Note として別 Issue 提案にとどめる
  ↓ No → 重大度判定へ
```

### ステップ2: 重大度の判定（高優先度）

**判断基準**:
指摘の影響範囲と緊急性に基づいて、Critical/Warning/Nit を判定。

**チェック項目**:

1. **Critical の判定**:
   ```
   ✅ セキュリティ脆弱性（SQLインジェクション、XSS等）
   ✅ データ損失リスク（トランザクション不備、バックアップ欠如）
   ✅ 本番障害に直結（デッドロック、無限ループ、メモリリーク）
   ✅ 法的リスク（ライセンス違反、GDPR 違反）
   ```

2. **Warning の判定**:
   ```
   ✅ バグの温床（レースコンディション、null 安全性欠如）
   ✅ 保守性の著しい低下（循環依存、God Class、密結合）
   ✅ 設計原則からの逸脱（SOLID 違反、関心の分離不足）
   ✅ テストの欠如（カバレッジ不足、境界値テストなし）
   ```

3. **Nit の判定**:
   ```
   ✅ 好みレベルの改善（変数名の冗長性、軽微なスタイル）
   ✅ 影響が限定的（局所的な可読性改善）
   ```

**判定フロー**:
```
指摘の影響を確認
  ↓
セキュリティ・データ損失・本番障害に直結するか？
  ↓ Yes → Critical
  ↓ No
  ↓
バグの温床・保守性低下・設計原則逸脱か？
  ↓ Yes → Warning
  ↓ No → Nit
```

### ステップ3: 技術品質と製品品質の分類（高優先度）

**判断基準**:
技術品質は製品品質に優先する。Summary でこの優先順位を明示。

**チェック項目**:

1. **技術品質の判定**:
   ```
   Security, Testing, Error Handling, Performance, Concurrency,
   Type Design, Data Modeling, Dependency, Architecture
   ```

2. **製品品質の判定**:
   ```
   Naming, Comments, Variables, Readability, SOLID, Functional,
   Component, API Design, State Design
   ```

**判定フロー**:
```
指摘の観点を確認
  ↓
技術品質か製品品質か？
  ↓ 技術品質 → 高優先度
  ↓ 製品品質 → 低優先度
```

### ステップ4: Good Points の抽出（中優先度）

**判断基準**:
指摘だけでなく、良い設計・実装を明示し、建設的なフィードバックにする。

**チェック項目**:

1. **良い設計の特定**:
   ```
   ✅ 適切な責務分離（レイヤー分離、関心の分離）
   ✅ 高いテストカバレッジ（境界値、異常系を網羅）
   ✅ 明確な命名（意図が伝わる変数名・関数名）
   ✅ 適切な抽象化（適度な共通化、DRY の実践）
   ```

2. **Good Points の書き方**:
   ```
   ✅ 具体的に何が良いか明示
   ✅ なぜ良いか理由を添える
   ✅ 学習効果を高める表現
   ```

### ステップ5: レポート形式の整備（低優先度）

**判断基準**:
読みやすく、アクションに繋がるレポート形式。

**チェック項目**:

1. **Summary の作成**:
   ```markdown
   ## Summary

   - Critical: N件（技術品質: N件 / 製品品質: N件）
   - Warning: N件（技術品質: N件 / 製品品質: N件）
   - Nit: N件
   ```

2. **指摘の構造化**:
   ```markdown
   ### [観点名] 指摘タイトル

   - **ファイル**: `path/to/file.ts:42`
   - **問題**: 何が問題か
   - **理由**: なぜ問題か
   - **提案**: どう修正すべきか
   ```

### ステップ6: 最終確認（低優先度）

**判断基準**:
レポート全体の一貫性と完成度を確認。

**チェック項目**:

1. **一貫性の確認**:
   ```
   ✅ 重大度が適切に判定されているか
   ✅ 技術品質と製品品質の区別が明確か
   ✅ 変更スコープ外の指摘が Note として扱われているか
   ```

2. **完成度の確認**:
   ```
   ✅ すべての指摘が観点名・ファイル・問題・理由・提案を含むか
   ✅ Good Points が含まれているか
   ✅ Summary が正確か
   ```

## レポート形式

```markdown
# Code Review Report

## Summary

- Critical: N件（技術品質: N件 / 製品品質: N件）
- Warning: N件（技術品質: N件 / 製品品質: N件）
- Nit: N件

## Critical

修正必須の問題。セキュリティ脆弱性、データ損失、本番障害に直結するもの。

### [観点名] 指摘タイトル

- **ファイル**: `path/to/file.ts:42`
- **問題**: 何が問題か
- **理由**: なぜ問題か
- **提案**: どう修正すべきか

## Warning

修正を強く推奨する問題。バグの温床、保守性の低下、設計上の問題。

### [観点名] 指摘タイトル

（同上の形式）

## Nit

好みレベルの提案。対応は任意。

### [観点名] 指摘タイトル

（同上の形式）

## Good Points

レビュー対象のコードで良かった点。
```

## 品質カテゴリ

レビュー観点は「技術品質」と「製品品質」の 2 つのカテゴリに分類される。

### 技術品質

その製品において「当然担保されている」と消費者が暗黙的に期待するもの。セールスポイントにはならないが、欠けたときの影響が広範囲に波及する（例: セキュリティ不祥事は企業の信頼そのものを毀損する）。ソフトウェアではテスト、セキュリティ、オブザーバビリティ、エラーハンドリング等がこれにあたる。

### 製品品質

その製品が生み出す価値。セールスポイントとなるもの。低くてもその製品が売れないだけで、影響は技術品質に比べると限定的。

### 優先順位

**技術品質は製品品質に優先する。** 技術品質に Critical が残っている状態で、製品品質の指摘を優先して対応してはならない。レポートの Summary でもこの優先順位を明示する。

### 観点の分類

| カテゴリ | 観点 |
|---------|------|
| 技術品質 | Security, Testing, Error Handling, Performance, Concurrency, Type Design, Data Modeling, Dependency, Architecture |
| 製品品質 | Naming, Comments, Variables, Readability, SOLID, Functional, Component, API Design, State Design |

## レビュー観点（詳細チェックリスト）

上記の判断プロセスを実行するための具体的なチェックリスト。

### 1. 変更スコープの確認

- [ ] レビュー対象の変更目的（新機能追加、不具合修正、hotfix、リファクタリング）を把握しているか
- [ ] 変更範囲を特定しているか
- [ ] 変更していない既存コードへの指摘を要修正（Critical/Warning）としていないか
- [ ] 既存コードの問題は Note として別 Issue 提案にとどめているか

### 2. 重大度の判定

- [ ] セキュリティ脆弱性・データ損失・本番障害に直結する問題を Critical としているか
- [ ] バグの温床・保守性低下・設計原則逸脱を Warning としているか
- [ ] 好みレベルの改善を Nit としているか
- [ ] 重大度が過大評価・過小評価されていないか

### 3. 技術品質と製品品質の分類

- [ ] 技術品質（Security, Testing 等）と製品品質（Naming, Readability 等）を正しく分類しているか
- [ ] Summary で技術品質と製品品質の件数を明示しているか
- [ ] 技術品質の Critical を製品品質より優先しているか

### 4. Good Points の抽出

- [ ] 良い設計・実装を特定しているか
- [ ] Good Points を具体的に記載しているか
- [ ] なぜ良いか理由を添えているか
- [ ] 指摘だけのレポートになっていないか

### 5. レポート形式の整備

- [ ] Summary が正確か（件数、技術品質/製品品質の区別）
- [ ] 各指摘が観点名・ファイル・問題・理由・提案を含むか
- [ ] 1 指摘 1 項目になっているか
- [ ] 指摘がない観点をレポートに含めていないか

### 6. 最終確認

- [ ] すべての指摘が変更スコープ内か
- [ ] 重大度が一貫しているか
- [ ] Good Points が含まれているか
- [ ] レポート全体が読みやすく、アクションに繋がる形式か

## レポート形式と指摘例

### Summary の例

```markdown
## Summary

- Critical: 2件（技術品質: 2件 / 製品品質: 0件）
- Warning: 5件（技術品質: 3件 / 製品品質: 2件）
- Nit: 3件
```

### Critical の指摘例

```markdown
## Critical

### [Security] SQLインジェクション脆弱性

- **ファイル**: `src/repositories/UserRepository.ts:42`
- **問題**: ユーザー入力を直接SQLクエリに埋め込んでおり、SQLインジェクションの脆弱性があります
- **理由**: `userId` がサニタイズされずにクエリに結合されているため、攻撃者が任意のSQLを実行できます
- **提案**: パラメータ化クエリを使用してください
  ```typescript
  // Before
  const query = `SELECT * FROM users WHERE id = ${userId}`;

  // After
  const query = `SELECT * FROM users WHERE id = ?`;
  db.execute(query, [userId]);
  ```
```

### Warning の指摘例

```markdown
## Warning

### [Testing] 境界値テストの不足

- **ファイル**: `tests/discount.test.ts:10`
- **問題**: 割引適用の境界値（10個）のテストがありますが、9個と11個のテストがありません
- **理由**: 境界付近で off-by-one エラーが発生する可能性があります
- **提案**: 以下の境界値テストを追加してください
  ```typescript
  test('should apply no discount for 9 items')    // 境界の直前
  test('should apply 10% discount for 10 items')  // 境界
  test('should apply 10% discount for 11 items')  // 境界の直後
  ```
```

### Nit の指摘例

```markdown
## Nit

### [Naming] 変数名の冗長性

- **ファイル**: `src/services/OrderService.ts:25`
- **問題**: 変数名 `orderData` が冗長です
- **理由**: `order` で十分に意図が伝わります
- **提案**: `order` に変更を検討してください
  ```typescript
  // Before
  const orderData = getOrder(orderId);

  // After
  const order = getOrder(orderId);
  ```
```

### Good Points の例

```markdown
## Good Points

- **適切な責務分離**: `OrderService` がビジネスロジックのみを持ち、DB アクセスは `OrderRepository` に分離されています。レイヤー間の依存方向も正しく、DIP（依存性逆転の原則）が適用されています
- **高いテストカバレッジ**: ユニットテストが正常系・異常系・境界値を網羅しており、テストコードも可読性が高いです。特に `UserService.test.ts` のテスト構造が参考になります
- **明確な命名**: 関数名と変数名が意図を明確に表現しており、コメントなしでもコードの意図が理解できます
```

## ルール

### 重大度の判定

| 重大度 | 基準 |
|--------|------|
| Critical | セキュリティ脆弱性、データ損失、本番障害に直結する問題 |
| Warning | バグの温床、保守性の著しい低下、設計原則からの逸脱 |
| Nit | 好みレベルの改善提案、軽微なスタイルの指摘 |

### 観点名

指摘の先頭に付ける観点名は以下のいずれか:

- Naming
- Comments
- Variables
- Readability
- SOLID
- Component
- Functional
- Error Handling
- Performance
- Security
- Testing
- Architecture
- API Design
- Type Design
- Data Modeling
- State Design
- Dependency
- Concurrency

### 指摘の書き方

- 1 指摘 1 項目。複数の問題をまとめない
- 問題・理由・提案の 3 点セットで書く
- 提案にはコード例を含めてよい
- 同じファイル・同じ行に複数観点の指摘がある場合もそれぞれ別項目にする

### Good Points

- レビュー対象で良い設計・実装があれば必ず記載する
- 指摘だけのレポートにしない

### レポート全体の方針

- 指摘がない観点はレポートに含めない（「問題なし」の羅列をしない）
- Critical が 0 件でも Summary には記載する
- すべての観点でレビューする必要はない。対象コードに関連する観点のみ選択する

### 既存コードの尊重

- どんなに設計上の問題があろうと、そのコードは過去に稼働し価値を生んできた。組織への合意なしに既存と異なるレイヤー構造やアーキテクチャを持ち込むべきではない
- 機能追加の文脈で部分最適を行うことはあるが、今回の変更に無関係な既存設計の問題を修正対象として指摘し、変更箇所を広げない
- 既存コードの設計上の問題に気づいた場合、問題の存在を認識させる目的で Note として言及することは構わないが、要修正の指摘（Critical / Warning）としては扱わない

### スコープの認識

- レビュー対象の変更の目的とスコープ（新機能追加、不具合修正、hotfix 等）を把握した上で指摘する
- hotfix や単純な不具合修正のレビューでは、変更されていない部分に対する設計上の問題点を要修正として指摘しない
- スコープを超えた指摘は、対応が必要な場合でも別の Issue やタスクとして切り出すことを提案するにとどめる

## テストケース

レビューレポート生成の精度を検証するためのテストケース。

### Case 1: 変更スコープ外の指摘を要修正としている（要修正）

**状況**: 新機能追加のレビューで、変更していない既存コードの設計問題を Critical で指摘

**期待される対応**:
- Critical/Warning ではなく Note として言及
- 別 Issue として切り出すことを提案
- 変更範囲内の問題のみを要修正として扱う

### Case 2: 重大度が不適切（要修正）

**状況**: SQLインジェクション脆弱性を Warning で指摘

**期待される対応**:
- セキュリティ脆弱性は Critical に格上げ
- 本番障害に直結する問題は Critical
- バグの温床は Warning

### Case 3: 技術品質と製品品質の区別がない（要修正）

**状況**: Summary で技術品質と製品品質を区別していない

**期待される対応**:
```markdown
## Summary

- Critical: 2件（技術品質: 2件 / 製品品質: 0件）
- Warning: 5件（技術品質: 3件 / 製品品質: 2件）
```

### Case 4: Good Points がない（要修正）

**状況**: 指摘だけを羅列し、Good Points がない

**期待される対応**:
- 良い設計・実装を特定して記載
- 具体的に何が良いか明示
- なぜ良いか理由を添える

### Case 5: 適切なレポート（指摘なし）

**レポート**:
```markdown
# Code Review Report

## Summary

- Critical: 1件（技術品質: 1件 / 製品品質: 0件）
- Warning: 2件（技術品質: 1件 / 製品品質: 1件）
- Nit: 1件

## Critical

### [Security] SQLインジェクション脆弱性

- **ファイル**: `src/repositories/UserRepository.ts:42`
- **問題**: ユーザー入力を直接SQLクエリに埋め込んでいます
- **理由**: `userId` がサニタイズされていません
- **提案**: パラメータ化クエリを使用してください

## Warning

### [Testing] 境界値テストの不足

（省略）

### [SOLID] 単一責任原則の違反

（省略）

## Nit

### [Naming] 変数名の冗長性

（省略）

## Good Points

- **適切な責務分離**: レイヤー間の依存方向が正しい
- **高いテストカバレッジ**: 正常系・異常系を網羅
```

**期待される判断**:
- 変更スコープ内の指摘のみ
- 重大度が適切
- 技術品質と製品品質を区別
- Good Points が含まれている
- 指摘なし
