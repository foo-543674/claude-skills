# パフォーマンスレビュー

コードレビュー時にパフォーマンスの観点で指摘すべきポイントを定義する。

## レビュー観点

### 1. N+1 問題

- ループの中で DB クエリや API コールを繰り返していないか
- 一括取得（batch fetch / eager loading）で解決できないか
- ORM の遅延読み込みが意図せず N+1 を引き起こしていないか

### 2. 不要な計算・重複処理

- ループ内で毎回同じ計算をしていないか（ループ外に移動できないか）
- 同じデータを複数回フェッチしていないか
- キャッシュすべき計算結果を毎回再計算していないか
- メモ化が有効な場面で使われているか

### 3. アルゴリズムと計算量

- O(n^2) 以上の計算量で、データサイズが大きくなる可能性があるか
- リスト探索を繰り返している場合、Map/Set への変換で O(1) ルックアップにできないか
- ソートや検索で適切なアルゴリズム・データ構造を使っているか

### 4. メモリ効率

- 大量データを一度にメモリに載せていないか（ストリーミング / ページネーションで処理可能か）
- 不要になったオブジェクトへの参照が残っていないか（メモリリーク）
- 大きな文字列の繰り返し結合で中間オブジェクトが大量に生成されていないか

### 5. I/O とネットワーク

- 逐次実行している I/O 操作を並行実行できないか
- 不要なデータまでフェッチしていないか（SELECT * の乱用、使わないフィールドの取得）
- レスポンスサイズが大きすぎないか（ページネーション、フィールドの絞り込み）

### 6. レンダリングパフォーマンス（UI）

- 不要な再レンダリングが発生していないか
- リストレンダリングに適切なキーが設定されているか
- 重い計算がレンダリングパスに含まれていないか（メモ化、Web Worker 等で回避）
- 画像やアセットの最適化（遅延読み込み、適切なサイズ）

### 7. 早すぎる最適化の回避

- パフォーマンスの問題が実際に発生しているか、計測に基づいているか
- 可読性を大きく損なう最適化をしていないか
- ホットパスかどうかの判断は適切か

## 指摘の出し方

- 「このループ内のクエリは N+1 になる。○○を事前に一括取得すれば 1 クエリで済む」のように改善案を具体的に示す
- 計算量の指摘では、想定されるデータサイズと実害の見積もりを添える
- 早すぎる最適化にならないよう、実際にボトルネックになりうるかを考慮する
