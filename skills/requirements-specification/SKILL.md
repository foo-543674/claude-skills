# 要件の仕様化

引き出した要件を、実装者がテストを書ける粒度まで具体化する。

このスキルのゴールは「要件の完了をどう検証するか」を定義すること。受け入れ基準が書けない要件は、まだ十分に具体化されていない。

## プロセス

### 1. ユーザーストーリーの記述

要件をユーザー視点の物語として記述する。

```
As a [ユーザー種別],
I want to [達成したいこと],
so that [得られる価値・目的].
```

**記述の品質基準（INVEST）**:
- **Independent**: 他のストーリーと独立して実装・リリースできる
- **Negotiable**: 実装方法は交渉・調整の余地がある
- **Valuable**: ユーザーに価値を提供する
- **Estimable**: 実装規模を見積もれる程度に具体的
- **Small**: 1 イテレーションで完了できるサイズ
- **Testable**: 完了を検証できる

大きすぎるストーリーは分割する。分割の軸はワークフローのステップ、データバリエーション、操作の種類（CRUD）など。

### 2. 受け入れ基準の定義

各ユーザーストーリーに対して、完了を判定するための基準を定義する。

**Given-When-Then 形式**:
```
Given [前提条件],
When [ユーザーのアクション],
Then [期待される結果].
```

**受け入れ基準の網羅性チェック**:
- 正常系: 主要なユースケースパスをカバーしているか
- 代替フロー: 正常だが異なるパスをカバーしているか
- 異常系: 不正入力、権限不足、リソース不足等のケースがあるか
- 境界条件: 入力値の上限・下限、空、null のケースがあるか
- 状態遷移: 対象リソースの状態によって振る舞いが変わるケースがあるか

### 3. 境界条件の明確化

曖昧さが残りやすい境界を具体的な値・条件で定義する。

**明確化すべき項目**:
- 入力値の許容範囲（最小値、最大値、文字数制限）
- リストの要素数制限（0件の場合の振る舞い、上限）
- 時間に関する境界（タイムゾーン、日付の切り替わり、有効期限）
- 同時操作（同じリソースを複数ユーザーが操作した場合）
- 権限境界（ロール別にできること・できないこと）

各境界条件に対して「その値のとき、システムはどう振る舞うべきか」を明記する。

### 4. 非機能要件の定量化

定性的な非機能要件を、測定可能な基準に変換する。

| 定性的 | 定量的 |
|--------|--------|
| 「速くレスポンスを返す」 | 「95パーセンタイルで 200ms 以内」 |
| 「大量データに対応」 | 「100万レコードでリスト表示 1秒以内」 |
| 「高可用性」 | 「月間稼働率 99.9%」 |
| 「安全に」 | 「OWASP Top 10 の各項目に対策済み」 |

定量化が難しい場合は、現時点での想定値と、その根拠を記録する。設計フェーズで検証・調整する前提で仮置きしてよい。

### 5. 画面・インタラクションの仕様（UI がある場合）

画面の仕様をワイヤーフレームレベルで定義する。

- 画面に表示する情報項目
- ユーザーの操作手順（フロー）
- 入力フォームのフィールドとバリデーションルール
- エラー時の表示・メッセージ
- 状態ごとの表示切り替え（ローディング、空状態、エラー状態）

ビジュアルデザインはここでは扱わない。情報と操作の仕様に集中する。

## 成果物

```markdown
# 仕様書: [機能名]

## ユーザーストーリー

### US-1: [ストーリータイトル]
As a [ユーザー種別],
I want to [達成したいこと],
so that [得られる価値].

**受け入れ基準**:
- [ ] Given [前提], When [操作], Then [結果]
- [ ] Given [前提], When [操作], Then [結果]

**境界条件**:
- [項目]: [条件と振る舞い]

## 非機能要件

| 項目 | 基準 | 根拠 |
|------|------|------|
| [項目] | [定量的基準] | [根拠・前提] |

## 画面仕様（該当する場合）

### [画面名]
- 表示項目: [リスト]
- 操作: [ユーザーアクション]
- 状態: [状態ごとの表示]
```

## ADR の作成基準

以下に該当する判断を行った場合、ADR を作成する:

- 受け入れ基準の定義で、ビジネスルールの解釈に複数の選択肢がある場合
- 境界条件の設定で、ユーザビリティとシステム制約のトレードオフがある場合
- 非機能要件の水準設定で、コスト・開発工数とのトレードオフがある場合

ADR のテンプレートと配置場所はプロジェクトの既存方針に従う。プロジェクトに ADR の慣習がない場合は `docs/adr/` に連番で配置し、MADR 形式を提案する。

## よくある失敗パターン

- **Vague Acceptance Criteria**: 「正しく動作すること」のような検証不能な基準
- **Missing Edge Cases**: 正常系の基準だけ定義し、異常系・境界を放置
- **Premature Design**: 仕様書に実装方法やDB構造を書いてしまう
- **Unmeasurable NFR**: 「十分なパフォーマンス」のような定量化されていない非機能要件
- **Overspecification**: UI の色やレイアウトまで仕様に含め、デザインの柔軟性を奪う
