# API設計レビュー

コードレビュー時にAPI設計の観点で指摘すべきポイントを定義する。REST に限らず、GraphQL や RPC スタイルにも適用できる共通原則を含む。

このスキルには筆者の設計思想が含まれる。絶対的なルールではなく、プロジェクトの文脈（API の公開範囲、クライアントの種類、チームの規模）に応じた判断が必要。

## Why: なぜAPI設計レビューが重要か

APIは、システム間の契約であり、一度公開すると変更が困難である。不適切なAPI設計は、長期的な技術的負債となる。

**根本的な理由**:
1. **契約の永続性**: 公開APIは、一度リリースすると変更が困難。クライアントが依存しているため、破壊的変更は大きなコストを伴う。初期設計の品質が、長期的な保守コストを決定する
2. **クライアント開発の効率**: 一貫性のないAPI（エンドポイントごとに異なるレスポンス構造、命名規則の不統一等）は、クライアント側の実装を複雑にし、バグの原因となる。統一された設計は、学習コストとミスを削減する
3. **パフォーマンスへの影響**: 不適切なAPI設計（over-fetching、chatty API、ページネーション欠如等）は、ネットワーク帯域とサーバー負荷を増大させ、ユーザー体験を悪化させる
4. **後方互換性の維持コスト**: バージョニング戦略が不明確だと、変更の影響範囲が予測不能になり、既存クライアントを壊すリスクが常につきまとう。明確なバージョニング戦略は、安全な進化を可能にする
5. **セキュリティリスク**: 内部実装の露出（DB構造がそのままレスポンスに現れる等）は、攻撃者に情報を与える。適切な抽象化は、セキュリティの一部である

**API設計レビューの目的**:
- 長期的に変更可能なAPI設計を保証する
- クライアント開発の効率を最大化する
- パフォーマンスとスケーラビリティを確保する
- 後方互換性を保ちながら進化できるようにする
- セキュリティリスクを最小化する

## 判断プロセス（決定ツリー）

API設計のレビュー時は、以下の順序で判断する。変更コストが高く、影響範囲が大きい問題から確認し、問題があれば指摘する。

### ステップ1: エンドポイント命名と構造の一貫性（最優先）

**判断基準**:
エンドポイント命名の不統一は、クライアント開発者の混乱を招き、学習コストを増大させる。

**チェック項目**:

1. **リソース名の統一**:
   ```
   ❌ 不統一
   GET /user/{id}      // 単数形
   GET /orders         // 複数形
   GET /product-list   // ハイフン区切り

   ✅ 統一
   GET /users/{id}     // すべて複数形
   GET /orders
   GET /products
   ```

2. **URL 命名規則の統一**:
   - kebab-case または snake_case を推奨（camelCase は環境依存のリスク）
   ```
   ❌ camelCase（避ける）
   GET /userOrders

   ✅ kebab-case
   GET /user-orders

   ✅ snake_case
   GET /user_orders
   ```

3. **RESTスタイルでのアクション動詞の排除**:
   ```
   ❌ RESTでアクション動詞
   POST /users/create
   POST /orders/delete

   ✅ HTTPメソッドで表現
   POST /users
   DELETE /orders/{id}

   【例外】RPC スタイル（gRPC, tRPC）ではアクション名がエンドポイント名になるのが自然
   ✅ RPC スタイル
   rpc CreateUser(...)
   rpc DeleteOrder(...)
   ```

4. **ネストの深さ**:
   - 3階層以上は要検討
   ```
   ❌ 深すぎるネスト
   GET /users/{userId}/orders/{orderId}/items/{itemId}/reviews

   ✅ 浅くする
   GET /order-items/{itemId}/reviews
   ```

**判定フロー**:
```
エンドポイント命名を確認
  ↓
プロジェクト全体で統一されているか？
  ↓ No → [Important] 既存エンドポイントとの一貫性を保つべき
  ↓ Yes
  ↓
RESTスタイルでアクション動詞を含んでいないか？
  ↓ Yes → [Important] HTTPメソッドで表現すべき
```

**判定**: 一貫性がない場合は**必ず指摘**（Important）

### ステップ2: リクエスト・レスポンス構造の一貫性（高優先度）

**判断基準**:
レスポンス構造がエンドポイントごとに異なると、クライアント側の実装が複雑になる。

**チェック項目**:

1. **Envelope パターンの統一**:
   ```typescript
   ❌ エンドポイントごとに異なる構造
   GET /users
   { users: [...] }

   GET /orders
   { data: [...] }

   ✅ 統一された構造
   GET /users
   { data: [...], meta: { total: 100 } }

   GET /orders
   { data: [...], meta: { total: 50 } }
   ```

2. **日時フォーマット、null の統一**:
   ```typescript
   ❌ 不統一
   GET /users
   { createdAt: "2024-01-01T00:00:00Z", tags: null }

   GET /orders
   { created_at: 1704067200, tags: [] }

   ✅ 統一
   GET /users
   { createdAt: "2024-01-01T00:00:00Z", tags: [] }

   GET /orders
   { createdAt: "2024-01-01T00:00:00Z", tags: [] }
   ```

3. **Over-fetching の回避**:
   - 不要なフィールドを返していないか
   - クライアントが使わないフィールドを大量に返していないか

**判定**: 構造の不統一は**必ず指摘**（Important）

### ステップ3: ページネーション・フィルタリング・ソート（高優先度）

**判断基準**:
リスト系エンドポイントで無制限にレコードを返すと、パフォーマンス問題とスケーラビリティ問題を引き起こす。

**チェック項目**:

1. **ページネーションの有無**:
   ```
   ❌ ページネーションなし
   GET /users
   // 10万件のユーザーを一度に返す

   ✅ ページネーションあり
   GET /users?page=1&limit=20
   GET /users?cursor=abc123&limit=20
   ```

2. **ページネーション方式の選択**:
   - **Offset ベース**: 「○ページ目に飛ぶ」要件がある（管理画面等）
   - **Cursor ベース**: リアルタイム性が重要、データ追加が頻繁
   ```
   【例外】Cursor が常に優れるわけではない
   管理画面の「3ページ目を見る」要件では offset が適切
   ```

3. **フィルタ・ソートの一貫性**:
   ```
   ❌ 不統一
   GET /users?filterByName=alice
   GET /orders?name_filter=alice

   ✅ 統一
   GET /users?filter[name]=alice
   GET /orders?filter[name]=alice
   ```

**判定**: ページネーションがない、方式が不適切な場合は**必ず指摘**（Important）

### ステップ4: エラーレスポンスの一貫性（中優先度）

**判断基準**:
エンドポイントごとにエラーレスポンスの構造が異なると、クライアント側のエラーハンドリングが複雑になる。

**チェック項目**:

1. **エラー構造の統一**:
   ```typescript
   ❌ 不統一
   POST /users
   { error: "Invalid email" }

   POST /orders
   { message: "Order not found", code: 404 }

   ✅ 統一
   POST /users
   { error: { code: "INVALID_EMAIL", message: "Invalid email" } }

   POST /orders
   { error: { code: "ORDER_NOT_FOUND", message: "Order not found" } }
   ```

2. **エラーコードの体系化**:
   - 機械的に処理可能なエラーコード（文字列）
   - HTTPステータスコードとの対応

**判定**: エラー構造の不統一は**指摘**（Minor）

### ステップ5: 冪等性と安全性（中優先度）

**判断基準**:
適切な冪等性設計がないと、リトライ時の二重実行やネットワーク障害時の不整合が発生する。

**チェック項目**:

1. **GET の副作用**:
   ```
   ❌ GET に副作用
   GET /users/{id}/increment-view-count
   // GETは副作用を持つべきではない

   ✅ POST で副作用
   POST /users/{id}/view-count
   ```

2. **PUT/DELETE の冪等性**:
   - 同じリクエストを複数回送っても結果が同じか

3. **POST の冪等性キー**:
   ```typescript
   ❌ 決済APIで冪等性キーなし
   POST /payments
   { amount: 1000 }
   // リトライ時に二重課金

   ✅ 冪等性キー
   POST /payments
   headers: { Idempotency-Key: "uuid" }
   { amount: 1000 }

   【例外】全てのPOSTに冪等性キーを求めるのは過剰
   二重実行で実害がある操作（決済、注文確定等）に絞る
   ```

**判定**: 冪等性が必要な操作で欠如している場合は**指摘**（Minor）

### ステップ6: バージョニング戦略（低優先度）

**判断基準**:
バージョニング戦略が不明確だと、破壊的変更の影響範囲が予測不能になる。

**チェック項目**:

1. **バージョニング方針**:
   - URL path: `/v1/users`
   - Header: `Accept: application/vnd.api+json;version=1`
   - Query: `/users?version=1`

2. **破壊的変更の定義**:
   - 非破壊: フィールド追加、オプションパラメータ追加
   - 破壊: フィールド削除・型変更、必須パラメータ追加

3. **例外の判断**:
   ```
   【例外】バージョニング不要なケース
   - 内部API（BFF等）でクライアントとサーバーが同時デプロイ
   - ビジネス上の移行期間が終了し、旧クライアントが存在しない
   ```

**判定**: 公開APIでバージョニング戦略が不明確な場合は**指摘**（Nit）

## レビュー観点（詳細チェックリスト）

- [ ] レスポンス構造がエンドポイント間で一貫しているか（envelope pattern 等）

- [ ] リスト系エンドポイントにページネーションがあるか
- [ ] ページネーション方式（offset vs cursor）が要件に合っているか
- [ ] フィルタ・ソートパラメータの設計が一貫しているか

### 4. エラーレスポンス

- [ ] エラーレスポンスの構造がエンドポイント間で一貫しているか
- [ ] 機械的に処理可能なエラーコードがあるか
- [ ] HTTPステータスコードが適切か

### 5. 冪等性と安全性

- [ ] GET が副作用を持っていないか
- [ ] PUT / DELETE が冪等か
- [ ] 二重実行で実害がある操作に冪等性キーがあるか（決済等）

### 6. バージョニング戦略

- [ ] 公開APIのバージョニング方針が決まっているか
- [ ] 破壊的変更の定義が明確か
- [ ] 旧バージョンの廃止計画があるか
- [ ] 内部API（BFF等）で不要なバージョニングをしていないか

## よくあるアンチパターン

- **Chatty API**: 1画面の表示に大量のエンドポイントを叩く必要がある（BFF や集約エンドポイントを検討）
- **RPC in REST Clothing**: REST を名乗りつつ `POST /api/getUsers` のようなエンドポイント
- **Inconsistent Error Shape**: エンドポイントごとにエラーレスポンスの構造が異なる
- **Leaking Internal Model**: 内部の DB 構造がそのままレスポンスに露出している

## 指摘の出し方

### 指摘の構造

```
【問題点】このAPI設計は○○の問題があります
【理由】△△だからです
【影響】この問題により××のリスクが発生します
【提案】□□に変更すると改善します
【文脈】ただし、◇◇の文脈では別の判断もあり得ます
```

### 指摘の例

#### エンドポイント命名の不統一の指摘

```
【問題点】このエンドポイント `/user/{id}` は、他のエンドポイントと命名規則が異なります
【理由】他のエンドポイント（`/users`, `/orders`）は複数形ですが、これだけ単数形です
【影響】クライアント開発者が混乱し、学習コストが増大します
【提案】以下のように複数形に統一してください
  ```
  // Before: 不統一
  GET /user/{id}
  GET /users
  GET /orders

  // After: 統一
  GET /users/{id}
  GET /users
  GET /orders
  ```
【文脈】プロジェクト全体で単数形に統一されている場合は、既存パターンに合わせてください
```

#### ページネーション欠如の指摘

```
【問題点】このリスト系エンドポイント `GET /users` にページネーションがありません
【理由】ユーザー数が増加すると、一度に10万件以上のデータを返すことになります
【影響】ネットワーク帯域とサーバー負荷が増大し、レスポンスタイムが悪化します
【提案】以下のようにページネーションを追加してください
  ```
  // Before: ページネーションなし
  GET /users

  // After: Offset ベース
  GET /users?page=1&limit=20

  // または Cursor ベース（リアルタイム性が重要な場合）
  GET /users?cursor=abc123&limit=20
  ```
【文脈】管理画面の「3ページ目を見る」要件がある場合は offset ベースが適切です
```

#### レスポンス構造の不統一の指摘

```
【問題点】このエンドポイントのレスポンス構造が他と異なります
【理由】`GET /users` は `{ users: [...] }` ですが、`GET /orders` は `{ data: [...] }` です
【影響】クライアント側の実装が複雑になり、バグの原因となります
【提案】以下のように envelope パターンを統一してください
  ```typescript
  // Before: 不統一
  GET /users
  { users: [...] }

  GET /orders
  { data: [...] }

  // After: 統一
  GET /users
  { data: [...], meta: { total: 100 } }

  GET /orders
  { data: [...], meta: { total: 50 } }
  ```
【文脈】BFF で特定の画面に最適化する場合は、この限りではありません
```

#### 冪等性キー欠如の指摘

```
【問題点】この決済APIに冪等性キーがありません
【理由】ネットワーク障害時のリトライで二重課金が発生する可能性があります
【影響】ユーザーへの過剰請求とビジネス上の問題となります
【提案】以下のように冪等性キーを追加してください
  ```typescript
  // Before: 冪等性キーなし
  POST /payments
  { amount: 1000 }

  // After: 冪等性キーあり
  POST /payments
  headers: { Idempotency-Key: "uuid" }
  { amount: 1000 }
  ```
【文脈】全てのPOSTに冪等性キーを求めるのは過剰です。二重実行で実害がある操作（決済、注文確定等）に絞ってください
```

### 指摘の優先度ラベル

- **[Important]**: 命名の不統一、レスポンス構造の不統一、ページネーション欠如（クライアント開発に直接影響）
- **[Minor]**: エラー構造の不統一、冪等性キー欠如、over-fetching
- **[Nit]**: バージョニング戦略の不明確さ（内部APIで影響小）

### 文脈依存の判断

API設計は文脈に強く依存するため、以下の文脈を常に考慮する：

- **API の公開範囲**: 公開API vs 内部API（BFF等）
- **クライアントの種類**: Webブラウザ vs モバイルアプリ vs サーバー間通信
- **デプロイ戦略**: クライアントとサーバーの同時デプロイ vs 独立デプロイ
- **チームの規模**: 小規模チーム vs 大規模・分散チーム
- **ビジネス要件**: リアルタイム性 vs バッチ処理、管理画面 vs エンドユーザー向け

## テストケース

レビュー判断の精度を検証するためのテストケース。

### Case 1: エンドポイント命名の不統一（Important指摘）

**コード**:
```typescript
// 既存のエンドポイント
GET /users
GET /orders

// 新しいエンドポイント
GET /product/{id}
```

**期待される指摘**:
- [Important] `/product/{id}` が単数形で、他（`/users`, `/orders`）と不統一
- `/products/{id}` に変更すべき

### Case 2: ページネーション欠如（Important指摘）

**コード**:
```typescript
// ユーザー数が10万件以上存在する可能性
GET /users
// ページネーションなし
```

**期待される指摘**:
- [Important] ページネーションがない
- `?page=1&limit=20` または `?cursor=abc&limit=20` を追加すべき

### Case 3: レスポンス構造の不統一（Important指摘）

**コード**:
```typescript
GET /users
{ users: [...] }

GET /orders
{ data: [...] }
```

**期待される指摘**:
- [Important] レスポンス構造が不統一
- envelope パターンを統一すべき

### Case 4: 冪等性キー欠如（Minor指摘）

**コード**:
```typescript
// 決済API
POST /payments
{ amount: 1000 }
// 冪等性キーなし
```

**期待される指摘**:
- [Minor] 決済APIに冪等性キーがない
- リトライ時の二重課金リスク
- Idempotency-Key ヘッダーを追加すべき

### Case 5: 許容されるケース（指摘なし）

**コード**:
```typescript
// BFF（Backend for Frontend）で画面に最適化
GET /dashboard/summary
{
  userCount: 100,
  orderCount: 50,
  recentOrders: [...],
  // 特定の画面に最適化されたレスポンス
}

// 内部API（クライアントと同時デプロイ）
GET /internal/cache-stats
// バージョニングなし
```

**期待される判断**:
- BFF は画面最適化が目的なので、構造の統一は不要
- 内部API（同時デプロイ）はバージョニング不要
- 指摘なし

### Case 6: RPC スタイル（指摘なし）

**コード**:
```typescript
// gRPC / tRPC スタイル
rpc CreateUser(CreateUserRequest) returns (User);
rpc DeleteOrder(DeleteOrderRequest) returns (Empty);
```

**期待される判断**:
- RPC スタイルではアクション名がエンドポイント名になるのが自然
- RESTの規則（アクション動詞の排除）を機械的に当てはめない
- 指摘なし
