# 可読性レビュー

コードレビュー時にコード構造・可読性の観点で指摘すべきポイントを定義する。

## レビュー観点

### 1. 関数の長さと責務

- 一つの関数が長すぎないか（目安として画面に収まらない関数は分割を検討）
- 関数内に複数の「段落」（空行で区切られた処理ブロック）がある場合、それぞれを関数に切り出せないか
- ただし「短くするために分割する」のではなく「意味のある単位で分割する」

### 2. ネストの深さ

- if/for/try のネストが 3 段以上になっていないか
- 深いネストは以下のテクニックで解消できないか:
  - 早期リターン / ガード節
  - 条件の反転（異常系を先に処理して正常系を浅く保つ）
  - 内側の処理を別関数に抽出

### 3. 早期リターン（ガード節）

- 事前条件のチェックを関数の冒頭でガード節として処理しているか
- 正常系と異常系が混在して読みにくくなっていないか
- else を書かずに済む構造にできないか

### 4. 制御フローの明快さ

- 条件分岐が複雑すぎないか（条件式に名前をつけて変数に切り出す検討）
- フラグ変数で制御フローを操作していないか（状態遷移が追いにくくなる）
- ループの中で break/continue が多用されて流れが読みにくくなっていないか

### 5. 処理の順序と流れ

- コードが上から下に読んで自然な順序になっているか
- 宣言と使用が離れすぎていないか
- 関連する処理がまとまっているか（無関係な処理が間に挟まっていないか）

### 6. 式の複雑さ

- 一行に詰め込みすぎていないか（メソッドチェーンが長すぎる、三項演算子のネスト等）
- 複雑な式は中間変数に分解して各ステップに名前をつけるべきか
- 論理式が複雑な場合、意味のある名前の変数や関数に切り出しているか

## 指摘の出し方

- 「この関数は○○と△△の2つのことをしている。分割すると個別にテストもしやすくなる」のように改善後のメリットを添える
- ネストの解消は具体的なリファクタリング手法（ガード節、関数抽出）を提案する
- 好みレベルのフォーマット指摘は避け、可読性に実質的な影響があるものに絞る
