# 依存関係設計レビュー

コードレビュー時に依存関係管理の観点で指摘すべきポイントを定義する。パッケージ・ライブラリの依存と、モジュール間の構造的な依存の両方を対象とする。

このスキルには筆者の設計思想が含まれる。依存関係は「追加は簡単、除去は困難」であり、慎重な判断を推奨する。ただし過度な依存回避（車輪の再発明）も保守コストを増やすため、バランスが重要。

## レビュー観点

### 1. ライブラリ選定基準

- メンテナンス状況は健全か（最終リリース日、Issue 対応速度、メンテナ数）
- ライブラリのサイズがバンドルサイズに与える影響を考慮しているか（フロントエンドの場合）
- ライセンスがプロジェクトの利用条件と互換性があるか
- 標準ライブラリや言語組み込み機能で代替できないか
- ただし十分にテストされたライブラリの再実装は保守コストが高い。「自前実装の品質 < ライブラリの品質」なら素直に使う

### 2. 依存方向と循環依存

- モジュール間の依存が一方向になっているか
- 循環依存が発生していないか（A → B → C → A）
- パッケージ・モジュールの境界を越える依存が最小限か
- 依存グラフが DAG（有向非巡回グラフ）になっているか
- ただしモジュール内の双方向参照は必ずしも問題ではない。パッケージ境界をまたぐ循環依存が問題

### 3. 外部依存の隔離と「捨てやすい構成」

- 外部ライブラリの API がアプリケーション全体に散在していないか。ラッパーを介してシグネチャや import 先の主導権をアプリ開発者がコントロールできるようにすることで、ライブラリが EOL になった際の差し替えを最小限の変更で実現できる（部分的なスクラップアンドビルドを可能にする「捨てやすい構成」）
- 外部ライブラリの型がドメイン層まで漏れていないか
- バックエンドでは DI コンテナやインターフェースによる抽象化の過程で自然に隔離される場合がある。一方フロントエンドは DI コンテナのような高度な依存性解決の仕組みがないため、ラッパーによる隔離のメリットが特に大きい
- ただしラップする範囲はケースバイケース。ライブラリの全機能を薄くラップするのではなく、アプリケーションが必要とするインターフェースに絞って隔離する

### 4. バージョン固定戦略

- lock ファイル（package-lock.json, Cargo.lock 等）がリポジトリにコミットされているか
- バージョン範囲指定（`^`, `~`）のリスクを理解しているか
- セキュリティアップデートの自動検出の仕組みがあるか（Dependabot, Renovate 等）
- メジャーバージョンアップの影響評価プロセスが定義されているか

### 5. 依存グラフの肥大化防止

- 小さな機能のために巨大な依存を追加していないか
- 推移的依存（transitive dependency）の量を把握しているか
- 同じ目的のライブラリが複数入っていないか（lodash と underscore の共存等）
- devDependencies と dependencies の区別が適切か
- ただしバンドルサイズの最適化はフロントエンドでは重要だが、バックエンドでは優先度が下がる場合がある

## よくあるアンチパターン

- **Left-Pad Problem**: trivial な機能のために外部依存を追加する
- **Full Surface Wrapper**: ライブラリの全 API を薄くラップしただけで、アプリケーション固有のインターフェース設計がない
- **Diamond Dependency**: 同じライブラリの異なるバージョンが推移的に依存される

## 指摘の出し方

- 依存追加のレビュー時は代替手段の有無と、依存を追加する理由を確認する
- ラッパーの提案は「捨てやすさ」（ライブラリ差し替え時の変更範囲の局所化）を根拠にする
- なぜその判断を推奨するかの理由を必ず添える
- 依存の削除を提案する場合は、代替実装のコストと品質も合わせて示す
