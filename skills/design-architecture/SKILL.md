# アーキテクチャ設計

要件をもとにシステムのアーキテクチャを設計する。

review-architecture の観点を設計時に先取りし、レビューで指摘されるような問題を事前に回避する。

## Why: なぜアーキテクチャ設計が重要か

アーキテクチャ設計の品質は、システムの長期的な保守性・拡張性・技術的負債の蓄積速度を決定する。不適切なアーキテクチャは、後から修正が極めて困難な構造的問題となる。

**根本的な理由**:
1. **変更容易性の決定**: アーキテクチャは変更の影響範囲を決定する。適切なレイヤー分離と境界定義により、変更が局所化され、リファクタリングが容易になる。不適切なアーキテクチャは、小さな変更が全体に波及し、修正コストが指数関数的に増大する
2. **技術的負債の蓄積速度**: 明確な境界がないアーキテクチャ（Big Ball of Mud）では、機能追加ごとに複雑さが蓄積し、技術的負債が指数関数的に増大する。適切なアーキテクチャは、技術的負債の蓄積を線形に抑える
3. **テスタビリティの確保**: 各レイヤーが独立してテスト可能なアーキテクチャは、テストコストを大幅に削減する。依存関係が混乱したアーキテクチャでは、テストに大量のモック・スタブが必要となり、テストの保守コストが膨大になる
4. **技術品質の組み込み**: セキュリティ・オブザーバビリティ・障害耐性等の技術品質は、アーキテクチャに組み込まれていなければ、後から追加が極めて困難。設計段階で組み込めば、自然に実現できる
5. **チームスケーラビリティ**: 明確な境界があるアーキテクチャは、チームが並行して開発できる範囲を広げる。境界が曖昧だと、変更の衝突が頻発し、チームの生産性が低下する

**アーキテクチャ設計の目的**:
- 変更の影響範囲を局所化し、変更容易性を確保する
- 技術的負債の蓄積を線形に抑え、長期的な保守性を維持する
- テスタビリティを確保し、品質を担保する
- 技術品質を設計段階から組み込む
- チームが並行して開発できる範囲を広げる

## 判断プロセス（決定ツリー）

アーキテクチャ設計時は、以下の順序で判断する。変更容易性とチームスケーラビリティに直結する判断から確認し、設計を進める。

### ステップ1: 既存アーキテクチャの把握（最優先）

**判断基準**:
既存アーキテクチャを無視した理想論は、移行コストと既存価値の破壊により失敗する。

**チェック項目**:

1. **既存構造の確認**:
   - ディレクトリ構造・モジュール分割の現状
   - 採用されているレイヤー構成（名前はプロジェクトごとに異なる）
   - 依存関係の方向
   - 既存の技術スタック・ADR

2. **継承 vs 刷新の判断**:
   - 既存アーキテクチャの問題点を特定
   - 段階的な改善で対処可能か、刷新が必要か
   - 移行コストと期待される効果のトレードオフ

**判定フロー**:
```
既存プロジェクトか？
  ↓ No → 新規設計へ
  ↓ Yes
  ↓
既存アーキテクチャを確認
  ↓
構造的な問題があるか？
  ↓ Yes → 段階的改善 or 刷新をADRで記録
  ↓ No → 既存構造を尊重して拡張
```

**判定**: 既存プロジェクトでは既存構造の理解と尊重を最優先

### ステップ2: レイヤー構成の設計（高優先度）

**判断基準**:
ドメインの本質・不変条件の複雑さ・進化の方向性に基づいてレイヤー構成を選択する。「今、CRUDで表現できるか」ではなく「CRUDで実現すべきか」を判断する。

**チェック項目**:

1. **ドメインの本質と不変条件の評価**:
   - **ドメインが守るべき本質は何か**:
     - 例: 銀行口座 → 「お金の動きを厳密に追跡する」
     - 表面的には「口座INSERT、残高UPDATE」で実現できる
     - しかし「UPDATE = 改ざん」となるため、本質的には不適切
     - → Immutableモデリング（入金・出金イベントの記録）が必要

   - **不変条件の複雑さ**:
     - 単純な制約（NOT NULL、UNIQUE等）→ DB制約で十分
     - 複雑な不変条件（複数エンティティにまたがる整合性、状態遷移の制約等）→ ドメイン層が必要
     - 監査要件（誰が、いつ、何をしたか記録）→ イベント駆動・Immutableモデリング

   - **ビジネスルールの性質**:
     - 単なるデータ保存・取得 → 簡易レイヤー
     - ビジネスの本質的な制約・ルール → ドメイン駆動設計

2. **進化の方向性の評価**:
   - **この先どう成長するか**:
     - 現在は単純でも、将来的に複雑なルールが追加される見通しがあるか
     - 例: 現在は「ユーザー登録・検索」だけだが、将来的に「サブスクリプション管理」「権限制御」「監査ログ」が追加される
     - → 初期から適切なレイヤー分離をしておく

   - **現在の要件に固執しない**:
     - 「今はCRUDで足りる」は、今後も足りるという意味ではない
     - ドメインの本質を見極め、成長に備えた設計を選択

3. **レイヤー設計の原則**:
   ```
   ビジネスロジック（安定）
         ↑
   アプリケーション層
         ↑
   インフラ層（不安定）
   ```
   - 安定した層に向かって依存する
   - 各レイヤーの境界を抽象で定義する

**判定フロー**:
```
ドメインの本質を評価
  ↓
守るべき不変条件は？
  ↓ 単純（DB制約で表現可能）
  ↓   かつ 将来の成長見通しがない
  ↓     → 簡易レイヤー（Controller-Service-Repository）
  ↓
  ↓ 複雑な不変条件（複数エンティティ、状態遷移等）
  ↓   または 監査要件がある
  ↓   または 将来の成長見通しがある
  ↓     → ドメイン駆動設計
  ↓
  ↓ Immutableモデリングが必要（履歴・監査）
  ↓     → イベント駆動アーキテクチャ
```

**判定**: 「今、CRUDで表現できるか」ではなく「ドメインの本質・進化を踏まえてCRUDで実現すべきか」で判断

### ステップ3: CQRS の適用判断（中優先度）

**判断基準**:
Command（書き込み）とQuery（読み取り）の要件が大きく異なる場合に分離する。

**チェック項目**:

1. **適用すべき兆候**:
   - 書き込みと読み取りで要求されるデータモデルが大きく異なる
   - 読み取りのパフォーマンス要件が書き込みと異なる
   - 複雑なビジネスルールと多様な検索条件が共存

2. **適用する場合の設計**:
   - Command側: 正規化、ビジネスルール厳守、トランザクション境界
   - Query側: 非正規化、読み取り最適化、結果整合性許容

**判定フロー**:
```
書き込みと読み取りで要求が大きく異なるか？
  ↓ No → CQRS不要（単一モデル）
  ↓ Yes
  ↓
パフォーマンス要件が異なるか？
  ↓ Yes → CQRS適用
  ↓ No
  ↓
将来的に分離が必要になる見通しは？
  ↓ Yes → CQRS適用を検討（ADRで記録）
  ↓ No → 現時点では不要
```

**判定**: 現在の要件で分離の必要性が明確な場合のみ適用

### ステップ4: 境界の定義（高優先度）

**判断基準**:
コンテキスト境界と外部システムとの接点を明確に定義し、変更の波及を防ぐ。

**チェック項目**:

1. **コンテキスト境界**:
   - 異なるドメインコンテキストを特定
   - 境界ごとに独立したモデルを持つ（同じモデルを共有しない）
   - 境界をまたぐデータ変換を一箇所にまとめる

2. **外部システムとの接点**:
   - 外部API・ライブラリとの接点に防腐層（Adapter/Anti-Corruption Layer）を設ける
   - 外部の変更が内部に波及しない設計

**判定**: 境界の明確化は変更容易性の確保に必須

### ステップ5: 技術選定（中優先度）

**判断基準**:
要件と制約、チームの状況を総合的に評価して選定する。

**チェック項目**:

1. **選定基準**:
   - 要件（機能要件・非機能要件）を満たせるか
   - チームの習熟度・学習コスト
   - コミュニティの成熟度・長期サポート
   - ライセンス・コスト制約
   - 既存技術スタックとの整合性

2. **トレードオフの記録**:
   - 技術選定の判断理由をADRに記録
   - 選ばなかった選択肢とその理由も明記

**判定**: トレードオフが発生した場合はADRで記録

### ステップ6: エラー境界の設計（中優先度）

**判断基準**:
レイヤー間でエラーを適切に変換し、ドメイン層に技術的詳細を漏らさない。

**チェック項目**:

1. **エラーの伝播方針**:
   - ドメイン層のエラーはドメインの言葉で表現（HTTPステータスコードではない）
   - Infrastructure層の例外がドメイン層まで漏れない
   - レイヤー境界でのエラー変換ルールを定義

2. **エラーの種類の列挙**:
   - ユースケースが返すエラーの種類を明確に定義
   - ビジネスルール違反 vs 技術的エラーを区別

**判定**: エラー境界の明確化はドメイン層の独立性確保に必須

## 基本原則

**技術品質を設計段階から組み込む**:

技術品質（テスタビリティ、オブザーバビリティ、セキュリティ、障害耐性等）は後付けではなく、アーキテクチャの設計対象として最初から扱う。これらは製品の機能ではないが、欠けたときの影響が広範囲に波及する。

設計時に以下を明示的に検討する:
- テスタビリティ: 各レイヤーが独立してテスト可能な構造か
- オブザーバビリティ: ログ・メトリクス・トレースの取得ポイントはどこか
- セキュリティ: 信頼境界はどこか、認証認可の責務はどのレイヤーが持つか
- 障害耐性: 外部サービス障害時の振る舞いをどう設計するか

## プロセス

### 1. 既存アーキテクチャの把握

新規プロジェクトでない限り、まず現在のアーキテクチャを理解する。

**確認すべきこと**:
- ディレクトリ構造・モジュール分割の現状
- 採用されているレイヤー構成（名前はプロジェクトごとに異なる）
- 依存関係の方向（どの層がどの層に依存しているか）
- 既存の技術スタック・フレームワーク
- 既存の設計判断の記録（ADR 等）

既存アーキテクチャを無視して理想のアーキテクチャを持ち込まない。既存の設計は過去に稼働し価値を生んできたもの。変更する場合は段階的に、合意を得て進める。

### 2. レイヤー構成の設計

変化の頻度と主導権に基づいてレイヤーを設計する。

**設計原則**:
- 変化の頻度が低い（安定した）層に向かって依存する
- ビジネスの主導権を持つ層が、技術的詳細に依存しない
- 各レイヤーの境界を抽象（インターフェース等）で定義する
- レイヤー間の結合点を最小限にする

**レイヤー設計の判断**:
- プロジェクトの複雑さに見合ったレイヤー数か（単純な CRUD にフルレイヤーは過剰）
- 各レイヤーの責務は明確か
- レイヤーをまたぐデータの受け渡し方法は決まっているか

### 3. CQRS の適用判断

Command（書き込み）と Query（読み取り）の分離が必要かを判断する。

**CQRS を適用すべき兆候**:
- 書き込みと読み取りで要求されるデータモデルが大きく異なる
- 読み取りのパフォーマンス要件が書き込みと異なる
- 複雑なビジネスルールを持つ書き込みと、多様な検索条件を持つ読み取りが共存する

**適用する場合の設計**:
- Command 側: クリーンアーキテクチャ等による適切なレイヤー分離、正規化されたモデル
- Query 側: 薄い構造、ストレージの能力を活かす設計、読み取り専用モデル

**適用しない場合**:
- 単純な CRUD で十分なコンテキストでは分離しない

この判断は ADR に記録する。

### 4. 境界の定義

コンテキスト境界と外部システムとの接点を設計する。

- 異なるドメインコンテキスト間の境界を明確にする
- 境界ごとに独立したモデルを持つ（同じモデルを共有しない）
- 外部 API・ライブラリとの接点に防腐層（Adapter / Anti-Corruption Layer）を設ける
- 境界をまたぐデータ変換を一箇所にまとめる

### 5. 技術選定

要件と制約に基づいて技術スタックを選定する。

**選定基準**:
- 要件（機能要件・非機能要件）を満たせるか
- チームの習熟度・学習コスト
- コミュニティの成熟度・長期サポート
- ライセンス・コスト制約
- 既存技術スタックとの整合性

技術選定でトレードオフが発生した場合は ADR を作成する。

### 6. エラー境界の設計

レイヤーをまたぐエラーの伝播方針を設計する。

- ドメイン層のエラーはドメインの言葉で表現する（HTTP ステータスコードではない）
- Infrastructure 層の例外がドメイン層まで漏れない設計にする
- レイヤー境界でのエラー変換ルールを定義する
- ユースケースが返すエラーの種類を列挙する

## ADR の作成基準

以下に該当する判断を行った場合、ADR を作成する:

- レイヤー構成の選択（Clean Architecture, Hexagonal, 簡易レイヤー等）
- CQRS の適用・不適用の判断
- 技術選定で複数の候補からの選択
- 既存アーキテクチャからの変更判断
- 境界の引き方で複数の選択肢がある場合

ADR のテンプレートと配置場所はプロジェクトの既存方針に従う。プロジェクトに ADR の慣習がない場合は `docs/adr/` に連番で配置し、MADR 形式を提案する。

## よくあるアンチパターン

- **Resume Driven Architecture**: チームの習熟度を無視して流行の技術を採用する
- **Big Ball of Mud**: 明確なレイヤー・境界なしに機能を追加し続ける
- **Overengineering**: 将来の拡張性を過度に意識し、現在の要件に対して複雑すぎる設計にする
- **Cargo Cult Architecture**: 有名企業の事例をそのまま模倣し、自プロジェクトの文脈を無視する
- **Framework Coupling**: ドメイン層にフレームワーク固有の依存を持ち込む
