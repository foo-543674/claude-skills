# 並行処理設計レビュー

コードレビュー時に並行処理・非同期処理の設計の観点で指摘すべきポイントを定義する。マルチスレッド、async/await、メッセージパッシング等、パラダイムを問わない共通原則を扱う。

このスキルには筆者の設計思想が含まれる。並行処理のバグは再現困難で発見が遅れやすいため、防御的な設計を推奨する。ただし過度なロックや同期は並行性のメリットを殺すため、リスクに見合った対策が必要。

## レビュー観点

### 1. レースコンディションとデータ競合

- 共有データへの同時アクセスが保護されているか（mutex, lock, atomic 等）
- Read-Modify-Write パターンで TOCTOU（Time of Check to Time of Use）の問題がないか
- DB の楽観的ロック（version カラム）や悲観的ロック（SELECT FOR UPDATE）が適切に使われているか
- ただしシングルスレッド環境（Node.js のイベントループ等）では CPU bound の共有データ競合は発生しない。ただし非同期 I/O の await 前後では論理的な競合が起こりうる

### 2. デッドロックと順序制御

- 複数のリソースをロックする場合、取得順序が統一されているか
- ロックの保持時間が最小限か（ロック内で I/O や長時間処理をしていないか）
- タイムアウト付きロックを使用しているか（無限待ちの防止）
- ただしデッドロック防止が過剰にロックを避ける方向に振れると、データ整合性が犠牲になる

### 3. 非同期処理パターンの適切性

- async/await のエラーハンドリングが漏れていないか（未処理の Promise rejection）
- 並行実行可能な処理を逐次実行していないか（`Promise.all` で並列化）
- 逆に、順序依存のある処理を不適切に並列化していないか
- バックプレッシャー（処理能力を超えるリクエストの流入）への対策があるか
- ただし全ての I/O を並列化すればよいわけではない。外部サービスのレート制限やコネクションプール枯渇を考慮する

### 4. トランザクション境界設計

- トランザクションの範囲が適切か（広すぎるとロック競合、狭すぎるとデータ不整合）
- 分散トランザクションが必要な場面で Saga パターンや補償トランザクションが検討されているか
- トランザクション内で外部 API 呼び出しをしていないか（ロック保持時間の増大）
- ネストされたトランザクションの振る舞いが意図通りか（SAVEPOINT の扱い）

### 5. リトライ・タイムアウト・バックオフ

- リトライ対象が冪等な操作に限定されているか
- 指数バックオフとジッターが実装されているか（thundering herd の防止）
- リトライ回数の上限が設定されているか
- タイムアウトが適切に設定されているか（上流のタイムアウト > 下流のタイムアウト）
- Circuit Breaker が必要な場面で導入されているか
- ただしリトライは問題を隠す可能性がある。根本原因の調査を怠らない

## よくあるアンチパターン

- **Fire and Forget**: 非同期処理の結果もエラーも確認しない
- **Lock Everything**: 過度なロックで並行性のメリットを完全に殺す
- **Async/Sync Mixing**: 同期処理の中で非同期処理を無理やり待機する（ブロッキング）
- **Unbounded Queue**: 上限のないキューがメモリを食い尽くす

## 指摘の出し方

- 並行処理の問題は再現が困難なため、問題が発生する具体的なシナリオ（タイミング）を示す
- 修正案は「なぜこの順序 / この方式で安全か」の根拠を添える
- 並行処理の複雑さとパフォーマンスのトレードオフを明示する
- なぜその設計を推奨するかの理由を必ず添える。「理論的にありうる」ではなく「実際に起こりうる条件」を示す
