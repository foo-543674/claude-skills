# skill-feedback

対話型スキルの利用後に foo-543674 から受けた指摘を、`foo-543674/foo-skills` の GitHub Issue として外部化し、プラグインの自己改善フィードバックループを回すスキル。

## Why: なぜ Issue として外部化するか

このリポジトリ（foo-skills）の存在目的は **foo-543674 の「複製」** である。思想を文書化し AI の知識で補完することで、最終的には AI に foo-543674 の仕事を代替させる。つまり対話中に発せられる「指摘」は、**複製がまだ十分でないことの兆候** であり、そのままスキル/エージェント定義の改善ネタとして吸い上げるべきものである。

ただし、このプラグインは **様々な端末・様々な作業リポジトリ** で起動された Claude Code から使われる。そこで感じた指摘をローカルのターミナル経由で foo-skills に反映するのは摩擦が大きく、フィードバックが失われやすい。

そこで **GitHub を中継点にした非同期フィードバックループ** を構築する：

```
作業リポジトリの Claude Code セッション
  └─ 対話型スキルを使用
  └─ 指摘が発生
  └─ /foo-skills:skill-feedback でバッチ Issue 化
        ↓
foo-543674/foo-skills に Issue 群
  └─ GitHub 上でトリアージ（採用 / 却下＋理由）
  └─ 採用 → スキル/エージェント定義の修正 PR（手動 or GitHub Actions）
        ↓
PR マージ → プラグイン更新 → 次回の対話で反映
```

**重要な思想**: Issue は TODO 管理ではなく **意思決定ナレッジベース** として使う。採用されなかった指摘も Close + 理由コメントで残すことで、将来「似たケースをどう判断したか」を検索可能にする。却下理由そのものが資産になる。

## スコープ

**含む**:
- 会話履歴から「指摘」の抽出と分類
- ログインユーザーの確認（foo-543674 本人のみ動作）
- 類似 Issue の検索と本文への参照追記
- `foo-543674/foo-skills` への Issue 作成

**含まない**:
- タスク完了の自動検知（明示的にこのスキルを呼び出す運用）
- Issue のトリアージ（GitHub 側で人間が行う）
- スキル修正 PR の自動生成（別途 GitHub Actions で扱う / 本スキルの責務外）
- foo-543674 以外のユーザーへのフィードバック送信

## 起動条件

- `/foo-skills:skill-feedback` による **明示的な呼び出しのみ**
- 原則として、対話型スキル（`requirements-*`, `design-*`, `plan-implementation`, `setup-plan` など）を使い終えた直後に呼ばれる想定

自動起動はしない。foo-543674 本人が「ここで区切って指摘を飛ばす」と判断したタイミングで叩く。

## ワークフロー

### Step 1: 認証ユーザー確認（ハードゲート）

ログインユーザーが `foo-543674` であることを確認する。本人以外が起動した場合は即座に中断する（他人のリポジトリに Issue を飛ばさない）。

ツール選択は以下の優先順位で判定する：

```
1. GitHub MCP が使えるか（mcp__github__get_me を試行）
   └─ 成功: レスポンスの login を確認
2. MCP が使えない場合、gh CLI で確認
   └─ `gh api user --jq .login` を実行
3. どちらも使えない
   └─ 「GitHub MCP も gh CLI も利用不可です。どちらかのセットアップが必要です」と表示して中断
```

`login != "foo-543674"` の場合：

```
このスキルは foo-543674 本人のフィードバックループ専用です。
現在のログインユーザー: <login>
Issue は作成しません。
```

と表示して終了する。

### Step 2: 会話履歴から指摘候補を抽出

直前までの会話履歴を見直し、`resources/detection-criteria.md` の判定規則に従って指摘候補を収集する。

抽出時の観点：
- **発話の前後関係**: foo-543674 の発話が、直前の AI の出力に対する修正・否定・言い換えになっているか
- **原因の所在**: その失敗の原因がスキル/エージェント定義で補える性質か（詳細は `detection-criteria.md`）
- **対象の特定**: どのスキル/エージェントのどの場面での失敗か

候補ゼロなら「発行する Issue はありません」と表示して終了。

### Step 3: 各候補を分類

各候補に対して以下の分類を行う（詳細は `resources/detection-criteria.md` と `resources/examples.md`）：

| 分類 | 処理 |
|---|---|
| ユーザー起因（入力省略・誤記・コンテキスト不足） | **除外** |
| 正当な対話設計の結果（段階的な確認、暗黙知の引き出し等） | **除外** |
| モデルの素の推論能力の限界（コンテキスト改善で補えない） | **除外** |
| 目的・前提の読み落とし | **confirmed** として採用 |
| コンテキスト改善で補えそう（グレーゾーン含む） | **candidate** として採用 |

**判定に迷ったら candidate にする**（全件 Issue 化の思想。判定の妥当性はトリアージ時に再検討される）。

### Step 4: 類似 Issue の検索

各採用候補について、`foo-543674/foo-skills` の既存 Issue を検索する：

```
1. GitHub MCP 優先: mcp__github__search_issues
2. フォールバック: gh search issues --repo foo-543674/foo-skills ...
```

検索クエリは対象スキル名 + 指摘キーワード（抽出された失敗の要点）で組み立てる。オープン/クローズ両方を対象にする（却下された類似判断も資産なので参照する）。

見つかった類似 Issue は本文の「類似 Issue」セクションに番号とタイトルを載せる。

### Step 5: Issue を発行

`resources/issue-template.md` のテンプレートに従って Issue 本文を組み立て、即座に発行する。**発行前の CLI 側プレビュー・承認ステップは挟まない**（トリアージは GitHub 側で行うため、CLI 側に意思決定を寄せない）。

ツール選択：

```
1. GitHub MCP 優先: mcp__github__issue_write (create)
2. フォールバック: gh issue create --repo foo-543674/foo-skills --title ... --body ... --label ...
```

ラベル：

| ラベル | 用途 |
|---|---|
| `feedback` | このスキル由来 Issue の目印（必須） |
| `confirmed` | 確定指摘 |
| `candidate` | グレーゾーン（却下時は Close + 理由で資産化される） |
| `skill:<name>` | 対象スキル/エージェント名（判明している場合のみ） |

### Step 6: 結果の要約表示

発行した Issue の一覧（番号・タイトル・ラベル・URL）を表示して終了する。

## AI の判断原則

- **発行前プレビューはしない**: 意思決定を CLI 側に寄せない。トリアージは GitHub の Web UI で行う前提。候補が見つかり次第、直接発行する。
- **候補ゼロも正常**: 指摘がなければ発行しない。無理に Issue を作らない。
- **迷ったら candidate で発行する**: false negative（取りこぼし）の方が損失が大きい。判定の妥当性は GitHub 側で再検討される。却下された事実と理由がナレッジになる。
- **ユーザー起因と正当振る舞いは厳格に除外する**: これらを混ぜると Issue のノイズが増え、トリアージ負荷が跳ね上がる。除外判定は保守的に（判断が本当に明確なときのみ除外、迷ったら採用側に倒す）。
- **対象スキル/エージェントを特定する努力をする**: Issue の `skill:<name>` ラベルは修正箇所の特定に直結する。会話中に使われていたスキル名が判然としない場合は、本文で「推定」と明記した上で記載する。
- **認証失敗は即中断**: ログインユーザーが foo-543674 でない場合、ツールが使えない場合、いずれも理由を短く表示して終了する。他人の環境で誤って Issue を飛ばさないための最優先ガード。
