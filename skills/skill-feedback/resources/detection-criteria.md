# 指摘検知の判定規則

会話履歴から「指摘」を抽出する際の採用/除外ルールと、分類（confirmed / candidate）の判定軸。

## 「指摘」の定義

**指摘 = スキル/エージェント定義（= コンテキスト設計）を改善すれば、次回から foo-543674 の期待通りに振る舞えるようになるもの**

この定義は Why に直結している。このリポジトリの目的は foo-543674 の「複製」であり、指摘はその複製がまだ十分でないことの兆候として扱う。**定義から外れるもの（ユーザー起因・正当振る舞い・モデル能力の限界）は Issue 化しない**。混ぜるとトリアージ負荷が跳ね上がり、ループが回らなくなる。

## 採用/除外の判定フロー

各候補発話に対して以下を順に確認する。

```
1. この失敗の原因は foo-543674 側にあるか？
   （入力省略、誤記、指示の曖昧さ、必要なコンテキストの未提供など）
   └─ Yes → 除外
   └─ No  → 2 へ

2. AI の振る舞いは、対話設計として正当なプロセスの一部だったか？
   （暗黙知の段階的な引き出し、確認のための問い返し、Phase に沿った進行など）
   └─ Yes → 除外
   └─ No  → 3 へ

3. その失敗はスキル/エージェント定義の改善で再発防止できそうか？
   （使えるツールの明示、判断基準の言語化、前提コンテキストの追加など）
   └─ Yes、明確に防げる          → confirmed として採用
   └─ グレー、補える可能性がある  → candidate として採用
   └─ No、モデル能力の限界        → 除外
```

**迷ったら candidate に倒す**。false negative（取りこぼし）の損失のほうが大きい。判断の妥当性は GitHub 側のトリアージで再検討される。

## 除外カテゴリの判定基準

### カテゴリ A: ユーザー起因

以下のいずれかに該当すれば除外。

- foo-543674 が自身の入力ミスや省略を明示的に認めている（「省いてた」「スマホ入力めんどくさくて」など）
- 直前のユーザー発話が曖昧・情報不足で、AI の解釈違いが発生していた
- 必要な前提情報（プロジェクト固有のコンテキスト等）が会話に提供されていなかった

判定が難しいケースも多いため、**「foo-543674 が明示的にユーザー起因だと述べた場合」以外は迷わず採用側に倒す**。勝手にユーザー起因と判定して除外するのは禁止。

### カテゴリ B: 正当な対話設計

以下のいずれかに該当すれば除外。

- 暗黙知を引き出すための段階的な質問（Phase 2-5 の Why / 意思決定プロセス / 比較 / エッジケース）
- 実装に入る前の確認ステップ（Phase 7 相当）
- スキル/エージェント定義で明示的に推奨されている振る舞い（例: 1 問ずつ聞く、発行前に確認する等）
- スコープ外の質問を避けるための確認

判定の手がかり：**そのスキル/エージェントの SKILL.md を読み直したとき、「これは定義通りの正しい動作」と言えるか？** 言えるなら除外。

### カテゴリ C: モデル能力の限界

以下のいずれかに該当すれば除外。

- コンテキストをどれだけ積んでも補えない推論能力の限界
- モデル固有の知識カットオフに起因する失敗
- スキル定義の改善で再発防止できる見込みがない

ただし、一見「推論能力の問題」に見えても、**コンテキスト設計で誘導できる余地があるもの**は candidate として採用する。例: 「使えるツールの一覧を明示していれば気付けた」「判断軸を列挙していれば選べた」など。

## 分類の判定軸（confirmed vs candidate）

### confirmed（確定指摘）

以下のいずれかが明確なら confirmed。

- **目的・前提の読み落とし**: リポジトリの目的、CLAUDE.md の方針、設計意図などを読めていない
- **スキル定義の欠落**: SKILL.md に書かれるべき判断基準・フロー・例が欠けており、同種の失敗が再発する
- **明示された振る舞いからの逸脱**: SKILL.md が「X せよ」と書いているのに X しなかった（これはスキル側の改善ではなく実装漏れだが、検知強化のための記述追加で防げる）

### candidate（候補）

以下のいずれかに該当するなら candidate。

- コンテキスト設計で補えるかどうか、その場で判定しきれない
- 推論能力の問題にも見えるが、プロンプト工夫で誘導できる可能性がある
- 同種の問題が再発する確信が持てない（単発のブレの可能性）

## 対象スキル/エージェントの特定

指摘を Issue 化する際、どのスキル/エージェントに対する指摘かを特定する。

判定手順：

1. **会話中に明示的に呼ばれたスキル名があればそれを採用**（`/foo-skills:<name>` で起動された場合）
2. 明示呼び出しがない場合、会話の内容から推定：
   - 要件定義的な問答 → `requirements-elicitation` または `requirements-specification`
   - 設計議論 → `design-architecture` / `design-api` / `design-data-model` / `design-component`
   - 実装計画 → `plan-implementation`
   - 環境構築の相談 → `setup-plan` 他
   - コードレビュー → `code-reviewer` (agent)
3. どうしても特定できない場合は、ラベルに `skill:<name>` を付けず、本文で「対象: 不明（会話履歴からは特定困難）」と明記する

## アンチパターン

検知時に避けるべき判断：

- **ユーザー起因だと勝手に決めつける**: foo-543674 が明示的にそう言わない限り、採用側に倒す
- **正当振る舞いと指摘を混同する**: SKILL.md の定義通りに動いた結果のユーザーリアクションを指摘と誤認する
- **1 指摘を複数 Issue に分解する**: 1 つの失敗パターンは 1 Issue にまとめる（関連する別の失敗は別 Issue）
- **指摘を 1 Issue に統合する**: 異なる失敗パターンを「まとめて Issue」にしない（修正 PR の粒度を壊す）
- **ラベル `skill:<name>` の推測を断定調で書く**: 推定なら推定と明記する
