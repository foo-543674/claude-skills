# 判定例

このスキル自体の設計対話（skill-feedback を作った最初の対話）を素材に、指摘検知・分類の判定を固定化した具体例集。回帰テスト代わりに使う。

新しい指摘パターンに遭遇して判断に迷ったとき、ここに書かれた類似例と比較して判定する。

> **注**: 以下の例は v1 時代（プラグインにスキル群が内蔵されていた構造）の対話を素材にしている。v2（ジェネレータ構造）では「スキル定義の改善」を「bootstrap 原料の改善」に読み替えて判定パターンを適用する。判定ロジック自体は同じ。

---

### 例 ①: 「このリポジトリの持ち主の名前見てください」

**場面**: Phase 1 で "foo" をプレースホルダと誤読し、"if the user is foo" を「ユーザーが何かを打ったら」と解釈していた場面で、foo-543674 がリポジトリオーナー名を見るよう指示した。

**AI の失敗（表面）**: "foo" の意味を推測で埋めようとし、GitHub コンテキスト（リポジトリオーナー名）を参照しに行く発想がなかった。

**判定: 除外（カテゴリ A: ユーザー起因）**

**理由**:
foo-543674 本人が「スマホ入力めんどくさくて省いてた。こちらの責任なので問題ない」と明示的にユーザー起因であると述べている。foo-543674 が "foo-543674" と正しく書いていれば発生しなかった。原料側の改善では防げない（「省略された入力を推測せよ」という指示を書くのは過剰）。

**教訓**: ユーザー起因かどうかの判定は、**foo-543674 本人の明示的な言及** を待つ。AI が勝手に「これはユーザー起因」と判断しない。

---

### 例 ②: 「違う gh コマンドでログインユーザー見ればいいでしょ」

**場面**: 例①でヒントをもらった後も推測を続けていた AI に対し、foo-543674 が具体的なツール名（gh コマンド）を提示した。

**AI の失敗（表面）**: `mcp__github__get_me` というツールが利用可能であったにもかかわらず、存在に気付かず推測を続けた。

**判定: candidate（カテゴリ: 原料改善で補える可能性あり）**

**理由**:
foo-543674 は「推論能力の問題だとも思うけど、どういうコンテキストを作り込むと推論能力を補完できるかわかるようなら取り込んでもいい」と述べた。純粋な推論能力の限界ではなく、**原料の改善（例: bootstrap テンプレートに「認証ユーザーの確認には mcp__github__get_me / gh コマンドを使う」という記述を含める）で誘導できる余地がある** と判断されたため、candidate として採用する。

**教訓**: 「推論能力の問題に見える」が「原料の工夫で補える可能性がある」ケースは candidate。トリアージ時に「補える」「補えない」が判断される。

### 例 ③: 「なんとなくズレてる。そもそもこのリポジトリの目的は私の複製なんですよ…」

**場面**: Phase 2 Why の深掘りで、AI が「A/B/C レイヤー（スキル自体への指摘 / 成果物への指摘 / 両方）」という分類を 2 ターン連続で提示していたところ、foo-543674 がリポジトリの本来の目的（複製プロジェクト）を説明し、AI の問いの立て方そのものが的外れだったことを指摘した。

**AI の失敗（表面）**: リポジトリの `.claude/CLAUDE.md` には目的が明記されているにもかかわらず、これを踏まえずに一般論ベースで問いを組み立てていた。

**判定: confirmed（カテゴリ: 生成物の品質不足 — 原料の欠落に起因）**

**理由**:
philosophy/core-principles に「語彙定義義務」が記述されているにもかかわらず、生成物のコンテキストにこのリポジトリの目的が十分に反映されていなかった。bootstrap テンプレートに「プロジェクトの目的・方針を最初に確認する」というステップを明示しておけば防げる。原料の改善で再発防止可能。

**教訓**: 生成された環境が「プロジェクトの目的を起点にした思考」を誘導できていない場合、bootstrap テンプレートまたは philosophy の改善で対処する。

### 例 ④: 「4 は暗黙知を引き出すというプロンプトの正当フローなので適切な振る舞い」

**場面**: AI が「④ 暗黙的な苛立ち — Issue 化の目的を自分から仮説立てできず、ユーザーに説明させてしまった」を指摘候補として挙げたが、foo-543674 はこれを指摘ではないと明言した。

**AI の失敗（表面）**: なし（むしろ「失敗だ」と AI が自己誤認していた）。

**判定: 除外（カテゴリ B: 正当な振る舞い）**

**理由**:
暗黙知を段階的に引き出すのは、生成された環境のコンテキストで推奨されている振る舞い。ユーザーに説明させる = 暗黙知を引き出している = 設計通りの正しい動作。これを「指摘」として Issue 化するとノイズになる。

**教訓**: 正当な振る舞いを「指摘」と混同しないこと。判定の手がかりは「**対応する原料ファイル（CLAUDE.md / エージェント定義 / 内蔵スキルの SKILL.md 等）を読み直したとき、これは定義通りの動作と言えるか？**」。言えるなら除外。

### 例 ⑤: 「自由記述で」と書きつつ選択肢を列挙してしまう（consult スキル違反 / v2 セッション）

**場面**: foo-543674 が consult スキルを使って相談中、AI が質問本文に「自由記述で」と注記しつつ、続けて「考えられる候補だけ列挙すると」と前置きして (i)/(ii)/(iii)/(iv) の 4 択を提示。foo-543674 から「なんでまだ選択肢出てくるの？」と即座に否定された。

**AI の失敗（表面）**: `skills/consult/SKILL.md` に「A/B/C/D や (i)〜(v) のような選択肢を AI が事前に列挙すると、ユーザーの思考をその枠に押し込めてしまう」と明記されているにもかかわらず、違反した。

**判定: confirmed（カテゴリ: 内蔵スキル原料の欠落に起因）**

**理由**:
スキル定義に違反したのは事実だが、違反検知の具体ヒューリスティック（番号付き候補列の禁止、「自由記述で」+ 列挙の自己矛盾パターンの明示、送信前セルフチェックの明文化）が原料に書かれていなかったため、原料の補強で再発防止できる見込みがある。bootstrap 生成物ではなく **plugin 内蔵スキルへの指摘だが、本スキルの対象に含まれる**（「foo-skills プラグインを通じてもたらされた振る舞いへの指摘」のため）。`source:skills/consult` ラベルは未整備のため省略し、本文の「対象」セクションで `skills/consult/SKILL.md` を明記する。

**教訓**: plugin 内蔵スキルへの指摘も対象。「bootstrap 生成物への指摘ではないから対象外」とは判定しない。`source:` ラベルが未整備の領域では新規ラベルを作成せず、本文に対象原料を記載する。

### 例 ⑥: 「pluginの中に入ってるスキルへのフィードバックを処理しないというのはありえない」（v2 セッション）

**場面**: AI が、上記 ⑤ のような plugin 内蔵スキル違反を「consult はジェネレータ側スキルで bootstrap 生成物ではないため scope 外」と判定して 0 件報告した。foo-543674 は「pluginの中に入ってるスキルへのフィードバックを処理しないというのはありえない」と否定し、スコープ解釈を明示的に修正した。

**AI の失敗（表面）**: skill-feedback の SKILL.md / detection-criteria.md / issue-template.md が揃って「bootstrap で生成された AI 自走環境への指摘」「bootstrap 原料の改善で防げる指摘」という文言で記述されており、AI がこれを文字通り解釈して plugin 内蔵スキルへの指摘を除外してしまった。

**判定: confirmed（カテゴリ: 原料のスコープ記述の欠落に起因）**

**理由**:
スコープ表現が bootstrap 生成物に過度に偏っていたのが原因で、原料側の修正（スコープ拡張）で再発防止可能。foo-543674 自身が明示的にスコープ解釈の修正を要求している。

**教訓**: 本スキルのスコープは「foo-skills プラグインを通じて foo-543674 の作業環境にもたらされた振る舞いへの指摘」であり、bootstrap 生成物に限定しない。plugin 内蔵スキル（consult 等）への指摘も等しく対象。

## この例集の使い方

1. 新しい指摘パターンに遭遇したとき、ここに書かれた例と比較する
2. 「どの例に似ているか」→「その例の判定」を参考にする
3. 既存の例と明らかに違うパターンが出てきたら、新しい例としてこのファイルに追記する（PR で提案）
