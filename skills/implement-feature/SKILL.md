# テストファースト機能実装

設計をもとに、テストファーストで段階的に機能を実装する。

## Why: なぜテストファーストが重要か

テストファースト（Test-First）の実践は、コードの品質・保守性・変更容易性を根本的に決定する。実装後のテスト追加では、テストが実装に引きずられ、設計の問題を見逃す。

**根本的な理由**:
1. **設計の明確化**: テストを先に書くことで、「何を作るか」を実装前に明確にする。曖昧な仕様・設計はテストを書こうとした時点で露呈する。実装を先に書くと、曖昧さを実装者の判断で埋めてしまい、後から仕様齟齬が発覚する
2. **インターフェース設計の改善**: テストを書く際に「使う側」の視点でインターフェースを設計する。実装を先に書くと、実装都合のインターフェースになり、使いにくい API が生まれる。テストファーストなら、使いやすいインターフェースが自然に設計される
3. **リグレッション防止の即時確保**: 実装と同時にテストが存在するため、リファクタリング・機能追加時のリグレッションが即座に検出される。実装後のテスト追加では、すでに技術的負債が蓄積し、テストを書く意欲が失われ、結果的にテストが書かれない
4. **テスタビリティの担保**: テストを書きながら実装することで、テスタブルな設計が強制される。実装を先に書くと、テスト困難な密結合・依存関係が生まれ、後からテストを書くコストが膨大になる。テストファーストなら、依存性注入・単一責任原則が自然に適用される
5. **技術的負債の予防**: Red-Green-Refactor サイクルにより、常にテストが通る状態で改善できる。実装を先に書き、後からテストを追加すると、テストなしで大量のコードが書かれ、構造的問題が固定化される。技術的負債の蓄積速度が指数関数的に増大する

**テストファーストの目的**:
- 仕様・設計の曖昧さを実装前に解消する
- 使いやすいインターフェースを設計する
- リグレッション防止を即時確保し、変更容易性を担保する
- テスタブルな設計を強制し、保守性を確保する
- 技術的負債の蓄積を防ぎ、長期的な開発効率を維持する

## 基本原則

**技術品質を先に作り込む**:

技術品質（テスト、エラーハンドリング、セキュリティ、オブザーバビリティ等）は、製品品質（機能の価値）を作る前に作り込む。「機能を作ってから技術品質を後付けする」のではなく、「技術品質が担保された土台の上に機能を作る」。

具体的には:
1. テストを先に書く（テストファースト）
2. エラーハンドリング・バリデーションを最初から組み込む
3. ログ・メトリクス等のオブザーバビリティを後回しにしない
4. セキュリティ要件を実装の最後ではなく設計段階から考慮する

**テストファースト（Red-Green-Refactor）**:
1. **Red**: 失敗するテストを書く（期待する振る舞いを定義する）
2. **Green**: テストを通す最小限のコードを書く
3. **Refactor**: テストが通る状態を維持しながらコードを改善する

テストは仕様の表現である。テストを先に書くことで「何を作るか」を実装前に明確にする。

## プロセス

### 1. 実装の準備

実装に入る前の確認事項。

**技術基盤の確認（新規リポジトリでは製品コードより優先）**:
- 開発環境: devcontainer が整備され、clone 後すぐに開発を始められるか
- CI/CD: テスト・lint・型チェックが自動実行され、マージブロックになっているか
- ローカルインフラ: DB 等の依存サービスがコマンド一発で起動できるか

これらが未整備の場合、製品コードの実装より先に整備する（setup-devcontainer, setup-ci, setup-local-infra）。

**前提の確認**:
- 要件仕様（受け入れ基準）が揃っているか
- 設計（アーキテクチャ、データモデル、API、コンポーネント）が決まっているか
- 既存コードベースの規約・パターンを把握しているか
- 開発環境でテストが実行できる状態か

**実装単位の分割**:
- 受け入れ基準を実装の単位に分割する
- 各単位が独立してテスト・コミットできるサイズか
- 依存関係が少ないものから着手する

### 2. テストファーストサイクル

1 つの実装単位ごとに以下のサイクルを回す。

#### 2.1 Red: テストを書く

受け入れ基準または設計仕様から、最初のテストを書く。

**テストの書き方**:
- テスト名で仕様を説明する（`should reject registration when email has no at sign`）
- Given-When-Then（Arrange-Act-Assert）で構造化する
- 1 テスト 1 アサーション（概念的に）
- まだ存在しないクラス・関数を呼び出すコードを書く（コンパイルエラーになる）

**テストの優先順位**:
1. 最もシンプルな正常系から始める
2. 次に代替フロー・バリエーション
3. 最後に異常系・境界値

テストを実行し、期待通りに失敗することを確認する。

#### 2.2 Green: テストを通す

テストを通す最小限のコードを書く。

**この段階のルール**:
- テストが通ることだけを目的にする
- コードの美しさ、設計の良さは後回し
- ハードコードでもよい（後のテストで一般化が強制される）
- ただし、設計で決めたレイヤー・責務の配置は守る

テストを実行し、すべてのテストが通ることを確認する。

#### 2.3 Refactor: コードを改善する

テストが通る状態を維持しながら、コードを改善する。

**リファクタリングの観点**:
- 重複の排除
- 命名の改善
- 関数・メソッドの分割（単一責任）
- 型の導入・改善
- 不要な複雑さの除去

**制約**:
- リファクタリング中にテストを壊さない
- リファクタリングと機能追加を同時にしない
- テストもリファクタリング対象にしてよい（可読性向上）

テストを実行し、すべてのテストが通ることを確認する。

### 3. サイクルの繰り返し

次の受け入れ基準・テストケースに対して、同じ Red-Green-Refactor を繰り返す。

**繰り返しの中で意識すること**:
- 前のサイクルのテストが引き続き通っているか（リグレッション防止）
- テスト間に実行順序の依存が生まれていないか
- テストの実行速度が許容範囲内か

### 4. 統合と最終確認

すべての実装単位が完了したら、統合的な確認を行う。

**最終確認項目**:
- 全テストが通るか
- 受け入れ基準をすべて満たしているか
- 設計で決めた構造・規約に従っているか
- 不要なデバッグコード・TODO が残っていないか
- コミット粒度は適切か

## テストの設計基準

review-testing の観点をテスト作成時に先取りする。

- **テスト名**: テスト実行結果に仕様を説明する文章が出力されること
- **網羅性**: 正常系・異常系・境界値をカバー
- **独立性**: テスト間に順序依存がないこと
- **テストダブル**: 外部依存の置き換えに限定。実装詳細のモック検証をしない
- **テストピラミッド**: ユニットテスト多数、統合テスト中程度、E2E 最少

## コード品質基準

review スキル群の観点を実装時に先取りする。

- **可読性**: 関数の長さ、ネストの深さ、早期リターン
- **命名**: 意図が伝わる名前、プロジェクトの命名規則
- **型安全**: 型ヒント・型注釈の使用、不正状態を型で排除
- **副作用の分離**: 純粋関数を優先し、副作用を境界に追い出す
- **エラーハンドリング**: ドメインの言葉でエラーを表現

## 実装完了チェックリスト

機能実装完了時に以下を確認する。

### テストファーストの実践確認
- [ ] すべての機能にテストが存在する（テストファースト）
- [ ] テストが失敗することを確認してから実装した（Red確認）
- [ ] テストがすべて通ることを確認した（Green達成）
- [ ] リファクタリングを実施し、コードの品質を改善した（Refactor実施）

### 受け入れ基準の達成確認
- [ ] 要件仕様の受け入れ基準をすべて満たしている
- [ ] 正常系・異常系・境界値のテストがある
- [ ] 不要なデバッグコード・TODOが残っていない

### コード品質の確認
- [ ] review-readabilityの観点: 関数の長さ、ネストの深さ、早期リターン
- [ ] review-namingの観点: 意図が伝わる名前、プロジェクトの命名規則
- [ ] review-type-designの観点: 型ヒント・型注釈、不正状態を型で排除
- [ ] review-functionalの観点: 純粋関数の優先、副作用の分離
- [ ] review-error-handlingの観点: ドメインの言葉でエラーを表現

### テスト品質の確認
- [ ] review-testingの観点: テスト名で仕様を説明、独立性、テストダブルの適切な使用
- [ ] テスト間に実行順序の依存がない
- [ ] テストの実行速度が許容範囲内

### 設計との整合性確認
- [ ] 設計で決めたレイヤー・責務の配置に従っている
- [ ] アーキテクチャ設計との一貫性
- [ ] コンポーネント設計との一貫性
- [ ] データモデル設計との一貫性

### コミット粒度の確認
- [ ] 意味のある単位ごとに小さくコミットした
- [ ] 複数の意味や目的を含むビッグバンコミットを避けた
- [ ] commit-messageのprefixルールに従った

## よくあるアンチパターン

- **Test After**: 実装を先に書き、後からテストを追加する（テストが実装に引きずられる）
- **Big Bang Implementation**: テストなしで大量のコードを書き、最後にまとめてテスト
- **Gold Plating in Green**: Green フェーズで必要以上にきれいなコードを書こうとする
- **Refactor Fear**: テストがあるのにリファクタリングを怖がり、Green で止まる
- **Testing Implementation**: 振る舞いではなく実装の詳細をテストする
- **Skipping Red**: テストが失敗することを確認せずに実装に入る
