# データモデル設計

要件をもとにデータモデルを設計する。

review-data-modeling の観点を設計時に先取りし、正規化・リレーション・履歴管理等の問題を事前に回避する。

## このスキルのスコープ

**扱う** (このスキルで決めること):
- エンティティと値オブジェクトの識別
- リレーション設計 (1:1 / 1:N / N:N、外部キー、削除カスケード)
- 正規化戦略 (どこまで正規化するか、非正規化の判断)
- **データ型の選定** (DATE / DECIMAL / TIMESTAMPTZ / VARCHAR(N) / JSONB 等の具体型)
- **直交する次元の格納方式** (独立カラム / 排他列挙 / 結合テーブル等)
- インデックス設計
- マイグレーション計画
- 履歴・監査テーブルの設計

**扱わない** (他のスキルに任せる):
- 集約のドメインメソッド・不変条件の置き場所 → `design-architecture` (の domain modeling)
- 永続化ポート (Repository) の配置層 → `design-architecture`
- リクエスト/レスポンス DTO の形状 → `design-api`
- ユーザーから見たセマンティクス → `requirements-specification`

スコープ外の項目は **出力しない**。テーブル定義に DDL を書くのは OK だが、その上でドメイン集約のメソッド一覧や Repository インターフェースを定義しない。低品質な出力は存在自体が害悪なので、書かない方がマシ。

## 基礎原則 (必読)

設計判断のすべてに以下 2 つの基礎原則を適用する。詳細は `../design-architecture/resources/principles.md` を必ず読むこと。

### 1. 語彙定義義務

テーブル名・カラム名・型名を「動詞 + 目的語」または「対象 + 性質」で 1 文定義できるようにする。`data` `info` `manager` `service` のような曖昧な名前を使わない。

### 2. 逃げの禁止と説明責任 (データモデル設計での典型逃げ)

#### 逃げ B: 数値的構造の文字列化

**問題**: 日付・金額・期間・座標など、数値の構造を持つデータを、parse やタイムゾーンの扱いが難しいという理由で文字列カラム (VARCHAR / TEXT) として持つ。

**なぜ駄目か**: 比較・計算・並び替え・範囲検索ができなくなり、本来 DB が担保してくれる境界条件のバグが頻発する。「問題を後段に押し付けただけ」で、後段の各所で同じ parse 処理を再発明することになる。

**代わりにすべきこと**:
- 日付 → DATE / TIMESTAMPTZ (要件に応じて TZ あり/なし)
- 金額 → DECIMAL(精度, スケール) (FLOAT は禁止)
- 期間 → INTERVAL またはミリ秒の INTEGER
- 座標 → 専用型 (PostGIS の GEOMETRY 等) または DECIMAL の組み合わせ

文字列で持つ判断を採用するなら、判断記録に以下を残す:
1. なぜ数値型ではダメか (制約・要因の具体)
2. 本当にそれ以外の方法がないか検討した経緯
3. 将来この制約が外れたら見直すトリガー

#### 逃げ C: 直交する次元の排他列挙化

**問題**: 「重量 + 回数」「時間 + 任意で重量」のように独立した属性の組み合わせを、`weight_reps` `time_only` のような排他列挙カラムで表現する。

**なぜ駄目か**: 組み合わせが増えるたびに列挙を拡張する必要があり、既存データのマイグレーションも発生する。「新しい組み合わせを追加できない」という設計的天井を作る。

**代わりにすべきこと**: 直交する次元 (重量 / 回数 / 時間 / セット数 / 符号 / etc.) を独立したカラムとして識別し、組み合わせは「どの属性が必須か」を表現するメタ情報 (別テーブル、JSONB の必須フラグセット等) で持つ。

排他列挙を採用する判断には、上記 1〜3 の説明責任を負う。

#### 逃げ F: 制約を型で表現せず Nullable で逃げる

**問題**: 「ある条件のときだけ必須」のような制約を、CHECK 制約や別テーブルで表現せず、すべてを Nullable にする。

**なぜ駄目か**: 「不正な状態」を型レベルで排除できず、アプリケーションコードに検証ロジックが散らばる。

**代わりにすべきこと**: 状態ごとに別テーブルに分ける、CHECK 制約で表現する、または STI (Single Table Inheritance) の代わりに Class Table Inheritance を使う。

## Why: なぜデータモデル設計が重要か

データモデル設計の品質は、システムの長期的な保守性・スケーラビリティ・データ整合性を決定する。不適切な設計は、後から修正が極めて困難な構造的問題となる。

**根本的な理由**:
1. **修正の困難さとコスト**: データモデルの変更は、マイグレーション・既存データの変換・アプリケーションコードの修正を伴い、コードのリファクタリングと比較して数十倍〜数百倍のコストがかかる。特に本番稼働後は、ダウンタイムやデータ損失のリスクを伴うため、修正が実質的に不可能な場合もある
2. **データ整合性への影響**: 不適切なモデリング（外部キー制約の欠如、Nullable の乱用、不正な状態の型レベル排除不足等）は、データ不整合を許容する構造となり、バグの温床となる。整合性が壊れたデータは、全体のデータ品質を低下させ、分析・レポート・ビジネス判断に影響する
3. **クエリパフォーマンスの決定**: テーブル構造・インデックス設計・正規化レベルは、クエリパフォーマンスを根本的に決定する。N+1 問題、不要な JOIN、フルスキャン等の性能問題は、モデル設計の段階で決まる。後からインデックス追加で解決できる範囲は限定的
4. **スケーラビリティの制約**: ロック競合を招く UPDATE 中心の設計、肥大化する単一テーブル、分散が困難な密結合なモデルは、スケールアウトの障壁となる。トラフィック増加時に対処不能な状況に陥る
5. **ビジネスロジックの複雑化**: 不適切なモデル（暗黙の状態遷移、Nullable による未確定状態の表現、ビジネス概念の不一致等）は、アプリケーション層のロジックを複雑にし、バグを誘発する。モデルが明確なら、ロジックはシンプルになる

**データモデル設計の目的**:
- 修正困難な技術的負債を初期段階で防ぐ
- データ整合性を型レベル・制約レベルで保証する
- クエリパフォーマンスとスケーラビリティを確保する
- ビジネス概念を正確に反映し、ロジックをシンプルにする
- 長期的な保守性と変更容易性を担保する

## 判断プロセス（決定ツリー）

データモデル設計時は、以下の順序で判断する。データ整合性とクエリパフォーマンスに直結する判断から確認し、設計を進める。

### ステップ1: エンティティの識別（最優先）

**判断基準**:
Entity（独自の識別子を持ち状態が変化する）とValue Object（識別子を持たず値で比較する）を明確に区別する。

**チェック項目**:

1. **Entity vs Value Objectの判断**:
   - 独立したライフサイクル（生成・変更・削除）を持つ → Entity
   - 値として比較され、変更されない → Value Object
   - 例: User → Entity、Email Address → Value Object

2. **集約の境界**:
   - トランザクション整合性を保証する範囲を決める
   - 集約ルート（Aggregate Root）を特定
   - 集約内部のEntityは集約ルート経由でのみアクセス

**判定フロー**:
```
要件から名詞を列挙
  ↓
独立したライフサイクルを持つか？
  ↓ No → Value Object（例: 住所、金額）
  ↓ Yes → Entity
  ↓
他のEntityと強い関係があるか？
  ↓ Yes → 同じ集約内のEntity
  ↓ No → 別の集約のルート
```

**判定**: EntityとValue Objectを明確に区別し、集約の境界を定義

### ステップ2: リレーション設計（高優先度）

**判断基準**:
参照整合性を保証するレベルと、パフォーマンスのトレードオフを判断する。

**チェック項目**:

1. **リレーションの種類**:
   - 1:N → 外部キー
   - N:M → 中間テーブル
   - 1:1 → 同じテーブル or 分割

2. **外部キー制約の判断**:
   - 同じ集約内 → 外部キー制約を付ける（強い整合性）
   - 異なる集約間 → ID参照のみ（弱い整合性、柔軟性重視）
   - カスケード削除は安易に使わない（明示的な削除処理を推奨）

3. **NULL許容の判断**:
   - NULLを避け、意味のあるデフォルト値や状態で表現できないか
   - Optional関係はNULL許容ではなく、別テーブルへの分離を検討

**判定フロー**:
```
エンティティ間の関係を特定
  ↓
同じ集約内か？
  ↓ Yes → 外部キー制約を付ける（強整合性）
  ↓ No
  ↓
参照整合性の厳密さは？
  ↓ 厳密に保証 → 外部キー制約（パフォーマンス犠牲）
  ↓ 柔軟性重視 → ID参照のみ（アプリ層で整合性担保）
```

**判定**: 集約境界と整合性要件に基づいて外部キー制約を判断

### ステップ3: 正規化と非正規化の判断（高優先度）

**判断基準**:
Command側（書き込み）は第3正規形、Query側（読み取り）は非正規化を検討。

**チェック項目**:

1. **正規化の原則（Command側）**:
   - 第3正規形を基本とする
   - データの重複を排除し、更新異常を防ぐ
   - 正規化のコストが利点を上回る場合のみ非正規化

2. **非正規化の検討（Query側）**:
   - 読み取りパフォーマンスのために必要な場合に限定
   - 非正規化したデータの同期方法を必ず設計
   - CQRSを採用している場合、Query側は非正規化が自然

**判定フロー**:
```
Command側 vs Query側？
  ↓ Command（書き込み）→ 第3正規形を基本
  ↓ Query（読み取り）
  ↓
読み取りパフォーマンスに問題があるか？
  ↓ No → 正規化のまま
  ↓ Yes
  ↓
非正規化のメリット > 同期コストか？
  ↓ Yes → 非正規化（ADRで記録）
  ↓ No → 正規化のまま
```

**判定**: Command側は正規化、Query側は必要に応じて非正規化（ADRで記録）

### ステップ4: 履歴・監査の設計（中優先度）

**判断基準**:
データの変更履歴が必要な場合は、Immutableモデリングを検討する。

**チェック項目**:

1. **履歴要件の確認**:
   - 変更履歴の追跡が必要か（監査要件、ビジネス要件）
   - 時点指定クエリが必要か（「XX日時点のデータ」）
   - 状態遷移の記録が必要か

2. **Immutableモデリング**:
   - 変更を上書きではなく、新しいレコードの追加として表現
   - 有効期間（valid_from, valid_to）で時点指定クエリを可能に
   - 状態遷移をイベントとして記録

3. **論理削除の判断**:
   - 履歴が必要 → 論理削除（deleted_at）
   - 履歴不要 → 物理削除

**判定フロー**:
```
変更履歴が必要か？
  ↓ No → updated_atで最終更新のみ追跡
  ↓ Yes
  ↓
時点指定クエリが必要か？
  ↓ Yes → Immutableモデリング（valid_from/valid_to）
  ↓ No → 論理削除（deleted_at）で履歴保持
```

**判定**: 履歴要件に応じてImmutableモデリングまたは論理削除を選択（ADRで記録）

### ステップ5: インデックス設計（中優先度）

**判断基準**:
クエリパターンを分析し、必要なインデックスを設計する。

**チェック項目**:

1. **インデックス候補の特定**:
   - WHERE句で頻繁に使われるカラム
   - JOIN条件で使われるカラム（外部キー）
   - ORDER BY、GROUP BYで使われるカラム

2. **インデックスの種類**:
   - 単一カラムインデックス
   - 複合インデックス（順序が重要）
   - ユニークインデックス（制約としても機能）

3. **インデックスのコスト**:
   - 書き込み時のオーバーヘッド
   - ストレージコスト
   - 過剰なインデックスは避ける

**判定**: クエリパターンに基づいて必要なインデックスを最小限設計

### ステップ6: マイグレーション戦略（低優先度）

**判断基準**:
既存データの移行と、スキーマ変更の安全な適用方法を計画する。

**チェック項目**:

1. **破壊的変更の段階的適用**:
   - カラム削除: 使用停止 → 削除の2段階
   - 型変更: 新カラム追加 → データ移行 → 旧カラム削除
   - テーブル分割: 新テーブル作成 → データコピー → 切り替え

2. **ダウンタイムの考慮**:
   - ダウンタイム許容 → シンプルなマイグレーション
   - ゼロダウンタイム → オンラインマイグレーション（複雑）

3. **ロールバック方法**:
   - マイグレーション失敗時の戻し方を事前に計画

**判定**: 破壊的変更は段階的に適用し、ロールバック方法を計画

## プロセス

### 1. 既存データモデルの把握

既存システムがある場合、現在のデータモデルを理解する。

**確認すべきこと**:
- 既存テーブル・コレクションの構造
- リレーション・外部キーの設定
- インデックスの設計
- マイグレーションの管理方法
- 既存の命名規則

### 2. エンティティの識別

要件から永続化が必要なエンティティを抽出する。

**識別の手順**:
1. 要件に登場する「名詞」を候補として列挙する
2. 各候補が独立したライフサイクル（生成・変更・削除）を持つか判断する
3. Entity（独自の識別子を持ち状態が変化する）と Value Object（識別子を持たず値で比較する）を区別する
4. 集約の境界を決める（トランザクション整合性を保証する範囲）

**命名規則**:
- テーブル名・カラム名はプロジェクトの既存規則に合わせる
- 既存規則がない場合は snake_case を推奨
- 省略せず、ドメインの用語をそのまま使う

### 3. リレーション設計

エンティティ間の関係を定義する。

**設計の判断基準**:
- 1:N か N:M か（中間テーブルの必要性）
- 外部キー制約を付けるか（参照整合性の保証レベル）
- カスケード削除の方針（安易にカスケードしない）
- NULL 許容の方針（NULL を避け、意味のあるデフォルト値や状態で表現できないか）

**集約をまたぐ参照**:
- 異なる集約間は ID による参照のみ（オブジェクト参照しない）
- 外部キー制約を張るかは、整合性要件とパフォーマンスのトレードオフ

### 4. 正規化と非正規化の判断

**正規化（Command 側の原則）**:
- 第3正規形を基本とする
- データの重複を排除し、更新異常を防ぐ
- 正規化のコストが利点を上回る場合のみ非正規化する

**非正規化（Query 側の最適化）**:
- 読み取りパフォーマンスのために必要な場合に限定する
- 非正規化したデータの同期方法を必ず設計する
- CQRS を採用している場合、Query 側は非正規化が自然

正規化 vs 非正規化の判断は ADR に記録する。

### 5. 履歴・監査の設計

データの変更履歴や監査ログの要件を設計に反映する。

**Immutable モデリング**:
- 変更を上書きではなく、新しいレコードの追加として表現する
- 有効期間（valid_from, valid_to）で時点指定クエリを可能にする
- 状態遷移をイベントとして記録する

**履歴が不要な場合**:
- updated_at カラムで最終更新のみ追跡すれば十分か判断する
- 論理削除（soft delete）の要否を検討する

### 6. マイグレーション戦略

既存データの移行と、スキーマ変更の安全な適用方法を計画する。

- 破壊的変更（カラム削除、型変更）はどう段階的に適用するか
- データ移行が必要な場合の手順とロールバック方法
- ダウンタイムの許容範囲と、オンラインマイグレーションの必要性

## ADR の作成基準

以下に該当する判断を行った場合、ADR を作成する:

- 正規化 vs 非正規化のトレードオフ判断
- 論理削除 vs 物理削除の選択
- Immutable モデリングの採用・不採用
- ID 生成戦略の選択（UUID, ULID, 連番等）
- 集約の境界設定で複数の選択肢がある場合

ADR のテンプレートと配置場所はプロジェクトの既存方針に従う。プロジェクトに ADR の慣習がない場合は `docs/adr/` に連番で配置し、MADR 形式を提案する。

## よくあるアンチパターン

- **God Table**: 1 テーブルにあらゆる属性を詰め込む
- **Polymorphic Association**: type カラムで参照先テーブルを切り替える（外部キー制約が効かない）
- **EAV (Entity-Attribute-Value)**: 柔軟性のために正規化を完全に放棄する
- **Premature Denormalization**: パフォーマンス問題が発生する前から非正規化する
- **Missing Index**: クエリパターンを考慮せずにインデックスを設計しない
- **Soft Delete Everywhere**: すべてのテーブルに論理削除を入れ、クエリが複雑化する
