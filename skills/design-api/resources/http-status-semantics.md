# HTTP ステータスコードのセマンティクス

API 設計時、HTTP ステータスコードは **プロトコル層の関心事に限定** する。ドメインバリデーションの細分類はステータスコードでは表現せず、レスポンスボディのエラーコードで表現する。

## 原則

1. **HTTP ステータスはプロトコル層の関心事に限定する**。ドメインの失敗理由はエラーコード (ボディ) で表現する
2. **ドメインバリデーションは原則 400 に集約する**。「バリデーションエラーは 400、ビジネスルール違反は 409」のような区別は作らない
3. **404 はパスパラメータ専用**。リクエストボディで参照された ID が存在しない場合は **400 (入力値の検証エラー)** として扱う
4. **409 は楽観ロック専用**。ユニーク制約違反・ドメインルール違反を 409 に寄せない
5. **「パスの not_found」と「ボディ参照の not_found」を命名で区別する** (例: `not_found` vs `parent_exercise_not_found` や `_in_body` 接尾辞)

## ステータスコード早見表

| ステータス | 正しい使いどころ | よくある誤用 |
|---|---|---|
| **400 Bad Request** | ドメインバリデーション全般。入力値が受け付けられない理由すべて (値の形式不正、ボディで参照された ID の不存在、既存値との衝突、ドメインルール違反、状態依存の拒否) | 「値の形式不正」だけに限定してしまう |
| **401 Unauthorized** | 認証トークンが無い / 無効 | 認可エラーに使う |
| **403 Forbidden** | パス対象は可視だが操作禁止 | 他人のリソースに使う (存在露呈するので 404 に寄せるべき) |
| **404 Not Found** | **パスパラメータが指すリソースの不存在のみ** + 他ユーザーのリソース (存在露呈防止) | ボディで参照された ID の不存在に使う |
| **409 Conflict** | **楽観的排他制御による並行更新コンフリクト専用** (ETag / バージョンカラムが噛み合わなかった場合) | ユニーク制約違反 / ビジネスルール違反に使う |
| **422 Unprocessable Entity** | RFC 的には構文は OK だが意味論的に処理できない。使うなら 400 との境界を明確化する必要あり。MVP では使わない選択も妥当 | 「バリデーションエラー」全般に使って 400 と混在させる |
| **429 Too Many Requests** | レート制限 | — |
| **500 Internal Server Error** | 予期しない内部例外 | `message` に内部詳細を出してしまう |

## よくある誤用パターン

以下はすべて **400 に集約すべき** 例である。

- `user_id_already_taken` → ❌ 409 / ✅ 400 (入力値の検証エラー)
- `exercise_name_conflict` → ❌ 409 / ✅ 400 (同上)
- `progression_cycle_detected` → ❌ 409 / ✅ 400 (ドメインルール違反)
- `progression_depth_exceeded` → ❌ 409 / ✅ 400 (同上)
- `measurement_set_locked_by_existing_records` → ❌ 409 / ✅ 400 (状態依存の検証エラー)
- ボディで参照された `parent_id` が存在しない → ❌ 404 / ✅ 400

## 判断基準

**「それはパスに含まれているか？」**
- Yes → 不存在なら 404
- No (ボディ・クエリ) → 不存在なら 400 (`*_not_found_in_body` 等の命名で区別)

**「それは楽観ロック (ETag / version) の噛み合わせ失敗か？」**
- Yes → 409
- No → 400 (ユニーク制約違反もドメインルール違反も 400)

**「それは認証の問題か、認可の問題か？」**
- 認証トークンなし・無効 → 401
- 認証済みだが操作禁止 → 403 (ただし他人のリソース存在露呈を避けるため 404 に寄せる選択もあり)
