# REST API と Web UI BFF の分離

アクション指向エンドポイント (ログイン・ログアウト等) と、リソース指向エンドポイント (CRUD) を **同じ名前空間に混ぜない**。PWA / SPA のようなフロントエンドを持つアプリでは、`/api/v1` (REST) と `/web/v1` (BFF for UI) の 2 つのパスファミリを持つことを検討する。

## 適用範囲: このドキュメントが対象とするアプリ

このドキュメントは **SPA / PWA のようなフロントエンドが、JavaScript からバックエンドのエンドポイントを fetch/XHR で叩くアーキテクチャ** を対象とする。`/web/v1` は「ブラウザの JS が裏で叩く BFF エンドポイント」のパスプレフィックスであって、**ブラウザのアドレスバーに表示されるページ URL のことではない**。

**対象外** (このドキュメントの分離方針を適用しない):

- **MPA (Multi-Page Application)**: テンプレートエンジンでサーバーがレンダリングする伝統的な Web アプリ。ページ URL はそのままユーザーに見える資産なので、`/web/v1/login` のような prefix は付けない。`/login` `/dashboard` `/users/:id` のような自然な URL を使う
- **テンプレートエンジンでフロントエンドとバックエンドが分離していないアプリ**: そもそも「BFF」という概念が存在しない。ページ URL = ルーティング
- **Server-side rendered フレームワーク (Rails, Django, Laravel 等) のページルート**: 同上

これらのケースでは、ページ URL の話と「外部公開する API があるか」の話が独立している。外部公開 API を持つなら `/api/v1` を別途用意する、持たないなら `/api` を作らないだけの話で、ページ URL に prefix を付ける議論は発生しない。

ブラウザのアドレスバーに `/web/v1/...` のような prefix が出るのはダサいし、そもそも MPA ではそれをやる動機もない。`/web/v1` は **ユーザーから見えない裏側の fetch エンドポイント** にだけ使う。

## Why: 動詞パスとリソースパスを混ぜてはいけない理由

初回の雑な設計では `POST /api/v1/auth/login`, `POST /api/v1/auth/logout`, `POST /api/v1/me/password` のような動詞パスを REST の `/api/v1` 名前空間に混在させがちだが、これは以下の観点で筋が悪い:

1. **意味論の不一致**: `login` / `logout` は「リソースを作る / 消す」より「認証フローを進める」という手続き的操作。REST の語彙に動詞パスが混ざると「REST っぽくない REST」になる
2. **外部公開時の語彙の歪み**: REST API を将来 3rd party に公開する場合、`login` のような Web UI セッション管理の語彙が露出してしまう
3. **認証モデルの拡張性**: REST API は将来 Bearer トークンを受け付ける余地があるが、Web UI 専用エンドポイントは Cookie 固定でよい。名前空間を分けておけばミドルウェアスタックを別にできる
4. **廃止・移行の独立性**: フロントエンドをリプレースしても REST はそのまま保てる (逆も可)

## 分離方針

**2 つのパスファミリを持つ**:

- `/api/v1/*` — REST リソース制御 (CRUD + 検索)。**リソース指向 URL のみ**
- `/web/v1/*`, `/mobile/v1/*` 等 — クライアント別の BFF。**動詞パス許容**、セッションライフサイクル管理

### BFF パスの命名: クライアント種別を明示する

BFF は要するに **「特定のクライアント専用の API ゲートウェイ」** である。したがって BFF パスのプレフィックスは **「誰向けの BFF か」** を明示しなければならない。

- ✅ `/web/v1` — Web フロントエンド (SPA/PWA) 向け
- ✅ `/mobile/v1` — モバイルアプリ向け
- ✅ `/admin/v1` — 管理画面向け
- ❌ `/bff/v1` — 「BFF である」ことしか言ってない。BFF はクライアント別に存在するものなので、「BFF」を名前にしても何の情報も伝わらない (`/api/v1` と同じくらい雑なラベリング)

クライアントが複数あるなら BFF も複数存在する。`/web/v1/me` と `/mobile/v1/me` は別のエンドポイントとして共存しうる (返すフィールドや認証方式が違う可能性がある)。`/bff/v1` という単一の名前空間に押し込めると、この区別ができなくなる。

## 典型的な振り分け

### `/web/v1` 側 (BFF / 動詞パス)

- サインアップ: `POST /web/v1/signup` — 自動ログイン + Cookie 発行というアクション面が強いため、`POST /api/v1/users` より BFF 側が妥当
- ログイン: `POST /web/v1/login`
- ログアウト: `POST /web/v1/logout`
- 現在ログイン中ユーザー情報: `GET /web/v1/me`
- パスワード変更: `POST /web/v1/change-password`

### `/api/v1` 側 (REST / リソース制御)

- ドメインリソースの CRUD (種目、セッション、ブロック、セット等)
- 検索・履歴取得
- ユーザーがリソースオーナーとして持つデータすべて

## 分離の判断基準

エンドポイントを `/web/v1` に置くか `/api/v1` に置くかは、以下の軸で判断する:

1. **そのエンドポイントは「リソースを操作する」のか「UI フローを進める」のか？**
   - リソース操作 → `/api/v1`
   - UI フロー (認証フロー、ウィザード、セッション管理) → `/web/v1`

2. **将来 3rd party クライアントに公開したい範囲に含まれるか？**
   - 含む → `/api/v1` (安定した REST 語彙を守る)
   - 含まない → `/web/v1` (フロントエンドの都合で自由に変えられる)

3. **Cookie セッションに依存するか、他の認証方式でも扱えるか？**
   - Cookie 固定でよい → `/web/v1`
   - Bearer トークンでも扱える必要がある → `/api/v1`

## 分離しない判断 = `/api` と名乗らない判断

単一のモノリシック Web アプリで、3rd party 公開の予定がなく、認証も Cookie 固定、REST 純度を保つ必要もない — このような場合は分離しなくてもよい。ただしその場合、**そのエンドポイント群を `/api/v1` と名付けるべきではない**。

**Why**: 「API」という語は外部から見た **インターフェース** を意味する。インターフェースの体をなしていない (リソース指向でもなく、動詞パスが混ざり、特定のフロントエンドの都合で形が決まっている) ものを `/api` と名乗ると、名前と中身が食い違う。外から見れば、そもそもバックエンドとフロントエンドに分かれていても **一つのアプリ** としか認識されないのだから、それを「API」と呼ぶのは虚偽のラベリングである。

**正しい選択**:
- インターフェースを公開しない → 全部 `/web/v1` (クライアント種別を表す名前) に統一する。`/api` を作らない
- インターフェースを公開する → `/api/v1` (REST 純度を守る) と `/web/v1` (BFF) を分ける

「`/api/v1` に動詞パスとリソースパスを混在させる」という選択肢は取らない。混在させた時点でそれは API ではなく BFF であり、BFF なら `/web/v1` と名乗るべき。

## アンチパターン

- **Mixed Verb and Resource Paths**: `/api/v1/users` と `/api/v1/auth/login` が同じ名前空間に混在している。REST 純度が下がり、外部公開時に動詞パスが露出する
- **BFF in REST clothing**: `/api/v1/auth/session` のような「REST っぽく見せかけた動詞パス」。リソースかアクションかをごまかしているので、判断基準の表明として弱い
