# テスト戦略・実装

プロジェクトまたは機能のテスト戦略を策定し、テストを実装する。

implement-feature がテストファーストで個別機能を実装するのに対し、このスキルはテスト全体の戦略設計と、既存コードへのテスト追加・改善を担う。

## プロセス

### 1. 既存テスト状況の把握

テスト戦略を策定する前に、現在のテスト状況を理解する。

**確認すべきこと**:
- テストフレームワーク・テストランナーの構成
- テストの配置規則（`__tests__/`, `*.test.ts`, `*.spec.ts` 等）
- 既存テストの粒度・種類（ユニット、統合、E2E）
- カバレッジの現状
- CI/CD でのテスト実行方法
- テストヘルパー・ファクトリの有無

### 2. テスト戦略の策定

テストピラミッドに基づいて、何をどの粒度でテストするかを決める。

**テストピラミッドの層**:

| 層 | 量 | 対象 | 実行速度 |
|---|---|------|---------|
| ユニットテスト | 最多 | ドメインロジック、ビジネスルール、データ変換 | 最速 |
| 統合テスト | 中程度 | コンポーネント間の連携、Repository + DB | 中 |
| E2E テスト | 最少 | ユーザーシナリオの主要パス | 最遅 |

**テスト対象の優先順位**:
1. ビジネスロジック（条件分岐、計算、状態遷移）— ユニットテスト
2. データアクセス層（Repository のクエリ）— 統合テスト
3. API エンドポイント（リクエスト → レスポンスの一連の流れ）— 統合テスト
4. ユーザーの主要操作フロー — E2E テスト

**テスト不要な対象**:
- フレームワークが保証する機能（ルーティング等）
- 単純な getter/setter
- 外部ライブラリの機能そのもの

### 3. テスト設計

テストケースを設計する。

**テストケース設計の手法**:
- 受け入れ基準からのテストケース導出
- 同値分割: 同じ振る舞いを期待する入力グループを特定し、各グループから代表値でテスト
- 境界値分析: 境界の前後でテスト（0, 1, 最大値, 最大値+1）
- 状態遷移: 状態マシンの各遷移パスをテスト
- デシジョンテーブル: 複数条件の組み合わせを網羅

**テスト名の設計**:
- テスト実行結果に仕様を説明する文章が出力されること
- 具体的な条件と期待結果を含む
- 良い例: `should reject registration when email has no at sign`
- 悪い例: `should return error`, `test validation`

### 4. テストコードの実装

設計したテストケースを実装する。

**テスト構造**:
```
Arrange: テストの前提条件をセットアップする
Act:     テスト対象を実行する
Assert:  結果を検証する
```

**テストダブルの使い方**:
- スタブ: 外部依存の戻り値を固定する（DB, API 等）
- モック: 相互作用の検証が本当に必要な場合のみ使う
- テストダブルの振る舞いが実際の依存先と乖離しないよう注意する
- モックが多すぎるテストは設計の問題を示唆する

**パラメータ化テスト**:
- 同じ構造で入力だけが異なるテストが並んだら、パラメータ化テストにまとめる
- ただし、テストケース間で Arrange が大きく異なる場合は個別テストの方が読みやすい

### 5. テスト品質の確認

実装したテストの品質を確認する。

**品質チェック**:
- テストが本当に失敗すべきとき失敗するか（対象コードを壊して確認）
- フレイキーテストの要因がないか（時刻依存、乱数依存、並行処理依存）
- テスト間に実行順序の依存がないか
- テストの実行速度が許容範囲内か
- テストコード自体の可読性・保守性

**カバレッジの考え方**:
- カバレッジは「テストされていない箇所の発見」に使う
- カバレッジの数値目標を達成すること自体を目的にしない
- 条件分岐カバレッジ（各分岐の真偽両方を通過）を重視する

## よくあるアンチパターン

- **Ice Cream Cone**: E2E テストが最も多く、ユニットテストが少ない（ピラミッドの逆転）
- **Testing Implementation**: 公開インターフェースではなく内部実装をテストする
- **Fragile Test**: 実装の微小な変更でテストが壊れる
- **Mystery Guest**: テストデータがテストファイルの外にあり、テスト単体で理解できない
- **Shared Fixture**: テスト間で可変な共有状態を使い、テストの独立性が損なわれる
- **Coverage Worship**: カバレッジの数値を上げることが目的になり、意味のないテストが増える
