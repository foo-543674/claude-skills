# 命名レビュー

コードレビュー時に命名の観点で指摘すべきポイントを定義する。

## Why: なぜ命名が重要か

命名はコードの可読性と保守性を決定する最も基本的な要素である。

**根本的な理由**:
1. **コードは書く時間より読む時間のほうが圧倒的に長い**: コードは一度書かれた後、何度も読まれ、理解され、修正される。命名が悪いと、毎回の読解に余計な認知負荷がかかり、チーム全体の生産性が継続的に低下する
2. **誤解はバグの温床**: 命名が誤解を招くと、実装者が意図と異なる使い方をし、バグが混入する。特に複数人が協業する場合、命名の誤解は連鎖的にバグを生む
3. **リファクタリングの起点**: 適切な命名は、コードの責務と抽象度を明確にする。不適切な命名は設計の問題を隠蔽し、技術的負債を蓄積させる
4. **ドメイン知識の共有手段**: 命名はチーム内でドメインの概念を共有する言語となる。不統一な命名はチームの認識のずれを生み、コミュニケーションコストを増大させる

**命名レビューの目的**:
- 将来のコード読解時の認知負荷を最小化する
- 誤解によるバグを未然に防ぐ
- チーム内の共通言語を育成・維持する
- 設計上の問題を早期に発見する

## 判断プロセス（決定ツリー）

命名のレビュー時は、以下の順序で判断する。優先度の高い問題から確認し、問題があれば指摘する。

### ステップ1: 誤解を招く命名か（最優先）

**判断基準**:
- 命名が実際の動作・役割と異なる印象を与えるか
- 単位・型・状態を暗示する命名が実態と一致しないか

**チェック項目**:
1. 関数名が副作用を隠していないか
   - ❌ `getName()` だが内部でAPIリクエストを発行している → ✅ `fetchName()`
   - ❌ `calculateTotal()` だが内部でDBを更新している → ✅ `updateAndCalculateTotal()`
2. 変数名が暗示する単位・型が実態と一致するか
   - ❌ `timeoutSec` だが実際にはミリ秒 → ✅ `timeoutMs`
   - ❌ `userList` だが実際には `Set` → ✅ `userSet` or `users`
3. 否定形の命名になっていないか
   - ❌ `isNotValid` → ✅ `isInvalid` or `!isValid`

**判定**: 誤解を招く命名は**必ず指摘する**（好みではなく正確性の問題）

### ステップ2: 意図が伝わらない命名か（高優先度）

**判断基準**:
- 命名だけを見て役割・目的が推測できないか
- 抽象的すぎる・汎用的すぎる名前か

**チェック項目**:
1. 抽象的すぎる命名
   - ❌ `process()`, `handle()`, `doSomething()` → ビジネス操作に置き換え
   - ❌ `Manager`, `Service`, `Helper`, `Util` → 具体的な責務を表す名前に
2. 汎用的すぎる命名
   - ❌ `data`, `info`, `result`, `tmp` → その処理での具体的な役割に
   - 例外: ループカウンタ `i`, ラムダの引数 `x` 等、慣習的に許容される短い名前
3. 略語の使用
   - チーム内で共通認識のある略語か
   - 許容例: `req`, `res`, `ctx`, `db`, `api`, `url` など広く通じるもの
   - 非許容例: プロジェクト固有の略語を説明なしに使用

**判定**: スコープが広い（public、モジュールエクスポート、クラスフィールド）場合は**必ず指摘する**。スコープが狭い（関数内ローカル変数）場合は文脈から推測可能なら許容する。

### ステップ3: プロジェクトの一貫性を破壊しているか（高優先度）

**判断基準**:
- 同じ概念に対して異なる単語を使っているか
- プロジェクト内の既存の命名パターンと異なるか

**チェック項目**:
1. 概念の一貫性
   - プロジェクト内で `user` と `account` が混在していないか確認
   - 既存コードで `User` と呼んでいる概念を `Account` と命名していないか
2. 動詞の一貫性（取得系）
   - プロジェクト内で `get`, `fetch`, `find`, `retrieve` のどれを使っているか確認
   - 既存パターンに合わせる（外部APIは `fetch`, DBは `find` など）
3. 真偽値の命名
   - `is`, `has`, `can`, `should` で始まるか
   - `true` がポジティブな状態を表すか（例外: HTML `disabled` など確立された慣習）
4. コレクションの命名
   - 複数形または `List`, `Set`, `Map` サフィックスがあるか

**判定**: プロジェクトの既存パターンを確認し、逸脱している場合は**必ず指摘する**。既存パターンがない場合は業界標準を推奨する。

### ステップ4: 文脈に応じた命名か（中優先度）

**判断基準**:
- 処理の文脈での役割を命名が表現しているか
- 型名をそのまま変数名にしていないか

**チェック項目**:
1. 型名の安易な再利用
   - ❌ `User user = getUser()` → この処理でのuserの役割は？
   - ✅ `User admin = getUser()` or `User recipient = getUser()`
2. 同じ型の複数の値の区別
   - ❌ `Date date1`, `Date date2` → 役割で区別
   - ✅ `Date startDate`, `Date endDate`

**判定**: 文脈から役割が明確でない場合、または同じ型の複数の値が区別しにくい場合は指摘する。

### ステップ5: 抽象度が適切か（中優先度）

**判断基準**:
- 命名の抽象度と実装の抽象度が一致しているか

**チェック項目**:
1. 具体的すぎる命名
   - ❌ `sendEmail()` だが実際にはSMS、プッシュ通知も送る → ✅ `sendNotification()`
2. 抽象的すぎる命名
   - ❌ `notify()` だが実際にはメール送信のみ → ✅ `sendEmail()`

**判定**: 将来の拡張を考慮しつつ、現時点の実装に見合った抽象度を推奨する。過度な抽象化（YAGNI違反）も指摘する。

### ステップ6: スコープに応じた長さか（低優先度）

**判断基準**:
- スコープの広さと命名の説明度が釣り合っているか

**スコープ別基準**:
- **public/export/クラスフィールド**: 説明的な名前が必須（3単語以上も許容）
- **関数内ローカル変数**: 簡潔でよい（1-2単語）
- **ラムダ・短いクロージャ**: 1文字も許容（`x`, `i`, `acc` など）

**判定**: スコープの広さに対して明らかに不十分な説明度の場合のみ指摘する。好みレベルの長さ調整は避ける。

## レビュー観点（詳細チェックリスト）

上記の判断プロセスを実行するための具体的なチェックリスト。

### 1. 誤解を招く命名のパターン

- [ ] 関数名が副作用を隠していないか（`get` だがAPIコール、`calculate` だがDB更新など）
- [ ] 変数名の単位が実態と一致するか（`Sec` だが実際は `Ms` など）
- [ ] 変数名の型が実態と一致するか（`List` だが実際は `Set` など）
- [ ] 否定形の命名になっていないか（`isNotXxx` など）
- [ ] 真偽値の `true` が予想と逆の意味になっていないか

### 2. 意図が伝わらない命名のパターン

- [ ] `process`, `handle`, `doSomething` など抽象的すぎる動詞を使っていないか
- [ ] `Manager`, `Service`, `Helper`, `Util` など責務が曖昧になるクラス名を使っていないか
- [ ] `data`, `info`, `result`, `tmp` など汎用的すぎる名前を安易に使っていないか
- [ ] 略語がチーム内で共通認識のないものでないか
- [ ] 一文字変数をループカウンタ以外で使っていないか

### 3. 一貫性を破壊するパターン

- [ ] 同じ概念に対して異なる単語を使っていないか（`user`/`account` の混在など）
- [ ] 取得系の動詞が統一されていないか（`get`/`fetch`/`find`/`retrieve` の混在）
- [ ] 真偽値が `is`, `has`, `can`, `should` で始まっていないか
- [ ] コレクションが複数形またはサフィックス（`List`, `Set`, `Map`）を持っていないか

### 4. 文脈に応じていないパターン

- [ ] 型名をそのまま変数名にしていないか
- [ ] 同じ型の複数の値が区別しにくい名前になっていないか

### 5. 抽象度が不適切なパターン

- [ ] 実装が汎用的なのに具体的すぎる名前がついていないか
- [ ] 実装が具体的なのに抽象的すぎる名前がついていないか

### 6. スコープに応じた長さの問題

- [ ] publicなのに短すぎる名前になっていないか
- [ ] ローカル変数なのに説明的すぎる長い名前になっていないか

## 例外ルール

以下の場合は、通常の命名ルールの例外として許容する。

### 許容される略語・短縮形

| 略語 | 意味 | 許容理由 |
|------|------|---------|
| `req`, `res` | request, response | Web開発で広く定着 |
| `ctx` | context | フレームワークで一般的 |
| `db`, `api`, `url` | database, API, URL | 技術用語として確立 |
| `i`, `j`, `k` | ループカウンタ | 数学・プログラミングの慣習 |
| `x`, `y`, `acc` | ラムダ引数 | 関数型プログラミングの慣習 |
| `e`, `err` | error | 短いエラーハンドリングブロックで一般的 |

### 許容される一文字変数

- ループカウンタ: `for (let i = 0; i < n; i++)`
- ラムダ・短いクロージャ（5行以内）: `items.map(x => x * 2)`
- 数学的文脈: `f(x, y)`, `Point(x, y)`

### 許容される否定形

- HTML属性など確立された慣習: `disabled`, `hidden`, `readonly`
- その処理で否定形のほうが明らかに可読性が高い場合（要理由説明）

### 許容される汎用名

- 明示的な一時変数: `const temp = a; a = b; b = temp;`（スワップ処理など）
- 汎用的なユーティリティ関数の引数: `clamp(value, min, max)`

## 指摘の出し方

### 指摘の構造

```
【問題点】この名前だと○○と誤解される可能性があります
【理由】△△だからです
【提案】□□に変更すると意図が明確になります
【影響】この変更により、××の改善が期待できます
```

### 指摘の例

#### 誤解を招く命名の指摘

```
【問題点】`getUser()` という名前ですが、内部でAPIリクエストを発行しています
【理由】`get` は副作用のない操作を期待させますが、実際にはネットワークIOが発生します
【提案】`fetchUser()` に変更すると、非同期操作であることが明確になります
【影響】呼び出し側で適切なエラーハンドリングやリトライ処理が考慮されやすくなります
```

#### 意図が伝わらない命名の指摘

```
【問題点】`OrderService` という名前ですが、責務が明確ではありません
【理由】Serviceという名前は何でも入る箱になりやすく、単一責任原則に反しやすいです
【提案】実際の責務を確認し、`OrderValidator`, `OrderPriceCalculator` など具体的な名前にしませんか
【影響】責務が明確になり、テストもしやすくなります
```

#### 一貫性を破壊する命名の指摘

```
【問題点】このプロジェクトでは `User` という概念を使っていますが、ここだけ `Account` と命名されています
【理由】同じ概念に異なる名前を使うと、別の概念と誤解される可能性があります
【提案】既存の命名に合わせて `User` に統一しませんか
【影響】チーム内の共通言語が維持され、コミュニケーションコストが削減されます
```

### 指摘の優先度ラベル

- **[Critical]**: 誤解を招く命名、バグの原因になりうる（必ず修正すべき）
- **[Important]**: 一貫性の破壊、意図が伝わらない（修正を強く推奨）
- **[Minor]**: スコープに応じた長さ、抽象度の微調整（修正を推奨）
- **[Nit]**: 好みレベルの調整（修正は任意）

### プロジェクトパターンの尊重

- 指摘前にプロジェクト内の既存コードを検索し、命名パターンを把握する
- 既存パターンがある場合は、それに従う提案をする
- 既存パターンが混在している場合は、より一般的なパターンを推奨し、将来的な統一を提案する

## テストケース

レビュー判断の精度を検証するためのテストケース。

### Case 1: 誤解を招く命名（Critical指摘）

**コード**:
```typescript
function getUser(id: string): Promise<User> {
  return fetch(`/api/users/${id}`).then(res => res.json());
}
```

**期待される指摘**:
- [Critical] `getUser` は副作用のない操作を期待させるが、実際にはネットワークIOが発生する
- 提案: `fetchUser` に変更

### Case 2: 意図が伝わらない命名（Important指摘）

**コード**:
```typescript
class UserService {
  process(data: any): any {
    // 複雑な処理
  }
}
```

**期待される指摘**:
- [Important] `UserService` の責務が不明確
- [Important] `process` が何をするのか不明確
- 提案: 具体的な責務に基づく命名

### Case 3: 一貫性の破壊（Important指摘）

**コード**:
```typescript
// 既存コードでは User を使用
const admin: User = findUser(adminId);

// 新規コード
const operator: Account = findAccount(operatorId);
```

**期待される指摘**:
- [Important] プロジェクト内では `User` を使用しているが、ここだけ `Account`
- 提案: `User` に統一

### Case 4: 文脈に応じていない命名（Minor指摘）

**コード**:
```typescript
function sendNotification(user: User) {
  const user2 = getRecipient();
  // user と user2 を使って処理
}
```

**期待される指摘**:
- [Minor] `user2` では役割が不明確
- 提案: 文脈に応じて `sender`/`recipient` など役割を表す名前に

### Case 5: 許容されるケース（指摘なし）

**コード**:
```typescript
const items = [1, 2, 3, 4, 5];
const doubled = items.map(x => x * 2);

for (let i = 0; i < items.length; i++) {
  console.log(items[i]);
}
```

**期待される判断**:
- ラムダの `x` は短いクロージャ内なので許容
- ループカウンタ `i` は慣習的に許容
- 指摘なし
